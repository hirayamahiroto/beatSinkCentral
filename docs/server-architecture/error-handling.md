# エラーハンドリング設計方針

各レイヤーのエラーハンドリング責務を明確に分離し、「どのレイヤーで何を検証/判定するか」を一貫させるための方針。

## エラーとは何か（メンタルモデル）

> **エラーとは、ある処理の実行中に、その処理が依拠しているルール（契約・前提条件・不変条件）への違反が検知されたことを、呼び出し側に報告するための値である。**

「想定外のことが起きた時に返されるもの」という受動的な捉え方では解像度が低い。エラーは「実行された処理において、**何のルールが、どう違反されたか**」を能動的に表明するものだと捉える。

### 定義の要素分解

- **実行中に** — 静的に気づける問題（型エラー等）とは別。ランタイムで検知される事象を扱う
- **依拠しているルール** — 明示的に定義されている必要がある。ルールがなければ違反もない。だからレイヤーごとにルールを持ち、エラーもレイヤーごとに定義される
- **違反が検知された** — 誰かが検知している。検知責任はそのルールを知っている主体（＝そのレイヤー）が持つ
- **呼び出し側に報告** — 受け取る相手がいる値。だから情報を載せる価値があり、捨てるべきでもない

### 正常系とエラーは対等な結果

処理を実行したときに起こりうる結末は次の2つ。

- ルールがすべて守られた → 正常系の結果を返す
- どこかでルール違反が起きた → どのルールが破れたかを報告する（エラー）

どちらも「実行した結果」であって、片方が主役で片方が例外ではない。エラーは正常系の裏面であり、**同じ実行の別結果**。関数のシグネチャを `Input → Output` ではなく `Input → Output | Error` と捉えると、エラーも必ず処理すべき分岐のひとつになる。

### ルールはレイヤーごとに異なる

| レイヤー       | そのレイヤーが持つルール | 違反時のエラー例                           |
| -------------- | ------------------------ | ------------------------------------------ |
| Value Object   | 値の形式ルール           | emailは@を含む、subIdは空でない            |
| Entity         | 状態の整合性ルール       | 公開状態なら必須項目が埋まっている         |
| Usecase        | 業務ルール（前提条件）   | 新規登録は未登録ユーザーに対してのみ行える |
| Infrastructure | 依存先との契約ルール     | DBは応答する、外部APIは時間内に返す        |

ルールを持つレイヤーがそのルールへの違反（エラー）を定義する。ルールを知らないレイヤーはそのエラーを投げられないし、投げるべきでもない。これが「エラーは発生したレイヤーで定義する」という原則の根拠。

### このモデルから導かれる帰結

- エラー名は「違反されたルールの名前」になる（例: `UserAlreadyRegisteredError` = 「新規登録の前提である未登録状態」への違反）
- エラーには「何の違反か」という情報が型として載るべき
- 握りつぶし = 違反の報告書を破り捨てる行為。「誰が悪かったか」「どう直すか」「再発するか」が全て失われる

## なぜこの設計が必要か

一言でいうと「**エラーを情報として扱い、失敗の原因と対処を呼び出し側が正しく判断できるようにするため**」。

### 解決したい課題

#### 1. 「どのレイヤー由来のエラーか」が判別できない問題

エラーが全て素の `Error` で投げられていると、受け取った側は以下を区別できない。

- 値が不正（ユーザーの入力ミス → 400で返すべき）
- 業務ルール違反（既に登録済み → 409で返すべき）
- 外部システム障害（DB接続失敗 → 500で返すべき）

結果、全部500で返すか、メッセージ文字列で `includes("already")` のような脆い判定をする羽目になる。型（`type` フィールド + 型ガード関数）で判別できるようにすることで、呼び出し側が安全に分岐できる。

#### 2. エラーの発生原因と対処責任の場所がぼやける問題

「既に登録済み」をドメインエラーとして投げると、ドメイン層がユースケース固有の事情（新規登録フローでしか意味がない制約）を知っていることになる。別のユースケース（例: 管理画面からの強制再同期）では「既に登録済み」はエラーですらないかもしれない。

エラーは **発生したレイヤー＝そのルールを知っているレイヤー** で定義する。レイヤー責務とエラー定義を一致させることで、「どこを直せばこのエラーが変わるか」が明確になる。

### エラーを握りつぶすと何が起きるか

握りつぶし = `catch (e) {}` や `catch (e) { return null }`、あるいは全エラーを一律500にマッピングする実装のこと。次のような問題が起きる。

1. **ユーザーに正しいフィードバックが返せない**
   「既に登録済み」なのに「サーバーエラー」と表示されると、ユーザーはリトライを繰り返す。本来は別のアクション（ログイン導線への誘導）が必要。

2. **本物の障害が埋もれる**
   DB接続失敗も業務ルール違反も同じ扱いになるので、監視・アラートで「いま本当に危険なエラー」を拾えない。500が1日1000件出ていても、そのうち何件が本物の障害か誰もわからなくなる。

3. **リトライ戦略が壊れる**
   一時的な障害（リトライで回復する）と恒久的な失敗（何度やっても同じ結果）を区別できないと、クライアントは無駄なリトライを繰り返すか、リトライすれば成功するケースを諦めてしまう。

4. **デバッグ情報が失われる**
   `catch (e) {}` で握りつぶすとスタックトレースも原因も消える。数ヶ月後に似たバグが出たとき、過去に同じエラーが起きていた事実すら残っていない。

5. **セキュリティ的な混乱**
   「認可違反（403）」と「リソース不存在（404）」を区別しないと、攻撃者に「存在するが権限がない」情報を与えてしまうケースや、逆にユーザーに本来返すべき情報を隠してしまうケースが混ざる。

### 本質

- 握りつぶし = **「失敗した」という最低限の事実しか残らない状態**
- 責務分離したエラー定義 = **「どのレイヤーの、どのルールに、どういう文脈で違反したか」を型に載せて伝える状態**

呼び出し側は型を見て分岐できるので、次を正しく判断できる。

- ユーザーに何を伝えるか
- リトライすべきか
- ログレベルをどうするか
- アラートを上げるか

これが「エラーを情報として扱う」ことの意味。

## 基本原則

- エラーは **発生したレイヤーで定義** し、そのレイヤーの責務範囲内の事象を表現する
- 下位層は上位層を知らない。エラーは型として上位層へ伝播する
- エントリポイント層で各レイヤーのエラーを捕捉し、HTTPレスポンスへマッピングする

## レイヤー別の責務

| レイヤー                      | 責務                                               | エラーの性質             | 例                                        |
| ----------------------------- | -------------------------------------------------- | ------------------------ | ----------------------------------------- |
| **Value Object（ドメイン）**  | 値の形式・制約の正当性検証                         | 値が不正                 | email形式不正、subId空文字                |
| **Entity（ドメイン）**        | エンティティ内の不変条件の検証                     | 状態が不正               | 必須関連の欠落                            |
| **Usecase**                   | ユースケース固有の前提条件・業務ルール             | 前提違反・業務ルール違反 | 既に登録済み、対象が存在しない            |
| **Entrypoint（APIハンドラ）** | 認証・認可・バリデーション・エラーのHTTPマッピング | プロトコル変換           | ドメインエラー→400、Usecaseエラー→409/404 |
| **Infrastructure**            | 外部システム起因の障害                             | 技術的障害               | DB接続失敗、外部APIタイムアウト           |

### ドメイン層（Value Object / Entity）

- **責務**: 値やエンティティそのものの正当性を保証する
- Value Object のファクトリ関数内で不正値を検知したら即時 throw する
- ドメイン層のエラーは「値・状態が不正」という性質のみを扱い、ユースケース固有の事情（既に登録済み等）には関与しない

### Usecase層

- **責務**: ユースケース固有の前提条件・業務ルール違反を表現する
- ドメイン層のエラーとは別に、各ユースケース配下にそのユースケース固有のエラークラスを定義する
- 「値の不正」ではなく「業務上その操作ができない状態」を表す
- 例: `UserAlreadyRegisteredError`（既に登録済みのため新規作成不可）

### Entrypoint層（APIハンドラ）

- **責務**: 認証・認可・入力バリデーション・エラーのHTTPマッピング
- Usecaseを呼び出し、`try/catch` で各レイヤーのエラーを捕捉する
- エラー種別に応じてHTTPステータスコードへマッピングする
- ビジネスロジックやDB操作は書かない

## 配置ルール

```
usecases/users/
├── createUser/
│   ├── index.ts           # ユースケース本体
│   ├── index.test.ts
│   └── errors.ts          # このユースケース固有のエラー
└── getMe/
    ├── index.ts
    └── index.test.ts
```

- エラークラスは **各ユースケース配下に co-located** で配置する
- ファイル名は `errors.ts`（複数エラーを1ファイルにまとめてよい）
- 共通基底クラス（`UseCaseError` 等）は **現時点では作らない**。複数ユースケースで同種の扱いが必要になった段階で導入を検討する

## 実装パターン（class不使用）

プロジェクト全体の方針（Entity/VOはtype + ファクトリ関数）に揃え、**エラーもclassを使わず `type + ファクトリ関数 + 型ガード関数` の3点セットで定義する**。

- **type**: `Error` を基底に、判別用の `type` フィールドと固有のコンテキスト情報を持つ型
- **ファクトリ関数**: `create{ErrorName}` 命名で `Error` インスタンスを生成し `Object.assign` でフィールドを付与
- **型ガード関数**: `is{ErrorName}` 命名で `error is {ErrorName}` を返す

`Error` を基底に使うのはスタックトレース・`throw` 互換性のため（JSの制約上、ここだけは `new Error()` が必要）。判別は `instanceof` ではなく **`type` フィールド + 型ガード関数** で行う。

## 実装例

### ユースケース固有エラーの定義

```typescript
// usecases/users/createUser/errors.ts
export type UserAlreadyRegisteredError = Error & {
  readonly type: "UserAlreadyRegisteredError";
  readonly subId: string;
};

export const createUserAlreadyRegisteredError = (
  subId: string
): UserAlreadyRegisteredError => {
  const error = new Error(
    `User already registered: ${subId}`
  ) as UserAlreadyRegisteredError;
  return Object.assign(error, {
    type: "UserAlreadyRegisteredError" as const,
    subId,
  });
};

export const isUserAlreadyRegisteredError = (
  error: unknown
): error is UserAlreadyRegisteredError => {
  return (
    error instanceof Error &&
    (error as Partial<UserAlreadyRegisteredError>).type ===
      "UserAlreadyRegisteredError"
  );
};
```

### ユースケース本体での throw

```typescript
// usecases/users/createUser/index.ts
import { createUserAlreadyRegisteredError } from "./errors";

export const createUserUseCase = async (input, userRepository) => {
  const existingUser = await userRepository.findBySub(input.subId);
  if (existingUser) {
    throw createUserAlreadyRegisteredError(input.subId);
  }
  // ...
};
```

### APIハンドラでの捕捉とマッピング

```typescript
// app/api/[[...route]]/users/create/index.ts
import { isUserAlreadyRegisteredError } from "../../../../../usecases/users/createUser/errors";

try {
  const result = await createUserUseCase(input, userRepository);
  return c.json({ userId: result.userId }, 201);
} catch (error) {
  if (isUserAlreadyRegisteredError(error)) {
    return c.json({ error: "User already registered" }, 409);
  }
  throw error;
}
```

## 層の独立性とは何か

エラーハンドリングもバリデーションも、根底にあるのは「層が独立している」という前提。この前提を言語化しておく。

### 各層は「異なる問いに答える専門家」である

「各層は責務が違う」という表現は正しいが、一段深くするとこう言える。

> **各層は、それぞれ固有の「問い」を持ち、他の層に代替不可能な答えを出す存在である。**

層ごとの問いを並べると次の通り。

| 層                     | 答えている問い                             | 失敗時の意味                         |
| ---------------------- | ------------------------------------------ | ------------------------------------ |
| **エントリポイント層** | このリクエストは受け取れる形で来ているか？ | 呼び出し側がプロトコルを守っていない |
| **Usecase層**          | この業務操作はいま実行してよいか？         | リクエストは正しいが業務状態的に不可 |
| **ドメイン層**         | この値・状態はビジネス概念として正当か？   | 値そのものが存在を許されない         |
| **Infrastructure層**   | 依存先は約束通り応答したか？               | 自分の責任ではない外部要因           |

これらの問いは**互いに独立**で、**1つを答えても他は答えたことにならない**。

### 誤解: 「層は同じ処理を段階的に厳密化している」

よくある誤った読み方:

> 「Zodで緩く見て、VOで厳しく見る。段階的に精緻化している」

これは間違い。正しい理解は次の通り。

> **ZodとVOは別々の事柄を別々にチェックしている。「非空」が両方に現れるのは、偶然の一致であって段階的精緻化ではない。**

- Zodが守っているのは **HTTPプロトコル上の約束**
- VOが守っているのは **ドメイン概念としての存在可能性**

これらは「同じ `email` フィールドに対する検証」に見えても、**守っている世界が違う**。Zodは「HTTPの世界」の守護者であり、VOは「ドメインの世界」の守護者。2つの世界が一瞬同じ文字列を経由するだけ。

### 責務とは「変更理由」である

単一責任原則の本質は「変更理由が1つであること」。この視点で層を再定義すると次のようになる。

| 層                 | 変更される理由                            |
| ------------------ | ----------------------------------------- |
| エントリポイント層 | HTTPプロトコル・APIスキーマが変わったとき |
| Usecase層          | 業務フロー・前提条件が変わったとき        |
| ドメイン層         | ドメインモデルのルールが変わったとき      |
| Infrastructure層   | 依存先のAPI・技術的実装が変わったとき     |

**変更理由が違う = 別のタイミングで変更される = 別の層にあるべき**。

「HTTPスキーマが変わっただけなのにドメイン層も直す必要がある」という状態は、責務が混ざっている証拠。逆に「ドメインルールが変わったのにHTTP層も直す必要がある」も同じ。

### 層は「知識の非対称性」を作る

クリーンアーキテクチャの核心として、層は**一方向の依存**を持つ。

- エントリポイント層はUsecaseを知っている
- Usecaseはドメインを知っている
- ドメインは何も知らない

この非対称性の意味はこうなる。

> **下位層（ドメイン）は、自分がHTTPから呼ばれていることを知らずに済むべき。そうあるからこそ、HTTP以外からも呼ばれ得る。**

ドメイン層が「自分はAPIリクエストの一部として呼ばれた」と知ってしまうと、CLIから呼んだときに挙動が変わってしまう。知らないからこそ、どこから呼ばれても同じ契約を守れる。

テストでも同じ。ドメインの単体テストはHTTPを知らずに書ける。HTTP層のテストはドメインの詳細を知らずに書ける。

### 層が「独立していること」の3つの意味

1. **問いが独立**: 各層は固有の問いに答える。他の層の問いを代行しない
2. **変更理由が独立**: 各層は別のタイミングで変更される。1箇所の変更が他に波及しない
3. **知識が独立**: 下位層は上位層を知らない。どこから呼ばれても同じ契約を守る

この3つが揃っているとき、層は「ただの整理のためのフォルダ分け」ではなく、**設計上意味のある境界**になる。逆にこの3つのどれかが崩れていると、層名だけがあって実態は混ざっている、という状態になる。

## 関連: バリデーションの責務分離

エラーハンドリングの設計と表裏一体の論点として、「バリデーションをどのレイヤーで実装するか」がある。結論から言うと、**HTTPリクエストのバリデーションとValue Objectのバリデーションは根本的に役割が異なるため、それぞれのレイヤーで独立して実装する**。

### 役割の違い

| バリデーション                              | 守っている契約                       | 責務の所在                         |
| ------------------------------------------- | ------------------------------------ | ---------------------------------- |
| **HTTPリクエストのバリデーション（Zod等）** | リクエストが期待する形式で来ているか | エントリポイント層（プロトコル層） |
| **Value Objectのバリデーション**            | 値自身がドメインの不変条件を満たすか | ドメイン層                         |

- HTTPバリデーションの検証対象は「リクエストボディ」というプロトコル上の構造物
- VOバリデーションの検証対象は「ドメインの値」というビジネス上の概念

両者は「たまたま似た検証（例: 非空チェック）をしている」ことはあっても、**守っている契約が別物**。同じルールが両方に書かれていても重複ではない。

### 具体例

「email は非空」というルールがZodとVO両方にあるケース。

- **Zodの非空** = 「HTTPリクエストに `email` フィールドが入ってきていること」
- **VOの非空** = 「`Email` 型として空文字は許されないこと（ドメインの不変条件）」

根拠が違うので、片方を消してはいけない。消すと「HTTP形式としては来ていないが、ドメイン値としては通ってしまう」あるいはその逆の穴ができる。

### クリーンアーキテクチャ的観点

エントリポイント層がドメイン層（VO）を直接importして使うこと自体は、依存方向が外→内なのでクリーンアーキテクチャ的には許容される。ただし**単一責任原則の観点**で見ると、次の問題が生じる。

- HTTP形式の検証とドメイン値の検証が1つのコードに混ざる
- CLI・バッチ・別プロトコルから同じUsecaseを呼ぶとき、HTTP層にベタ書きしたロジックは再利用できない（Usecase内のVO検証は再利用できる）
- テストの責務も混ざる（HTTP層のテストでドメインルールの網羅が必要になる）

したがって「可能かどうか」ではなく「責務として分けるべきか」で判断する。**分ける**が答え。

### 採用する設計

| 層                     | 責務                                               | 使うもの                          |
| ---------------------- | -------------------------------------------------- | --------------------------------- |
| **エントリポイント層** | HTTPリクエストの形式検証（必須・型・最低限の構造） | Zod                               |
| **Usecase層**          | 受け取った生の値をVOに変換しドメイン検証を実行     | ドメインのVOファクトリ            |
| **Value Object**       | 値自身のドメイン不変条件                           | 自前の検証ロジック（内部でzod可） |

### 実装ポリシー

1. **Zodスキーマは「HTTPリクエストとしての形式」だけを書く**
   - 必須・型・最低限の長さ等、HTTPプロトコル上の制約に限定する
   - ドメインのルール（emailの形式、accountIdの命名規則等）までは踏み込まない
2. **Usecaseは生の値（string等）を受け取り、内部でVOに変換する**
   - VO変換時のバリデーションエラー = ドメインエラーとして上位に伝播
   - これによりUsecaseが「ドメイン契約を守る最後の砦」になる
3. **VOをHTTP層から直接呼ばない**
   - エントリポイント層がドメイン層を直接参照する形は避ける
   - 依存は `エントリポイント → Usecase → ドメイン` の一方向に保つ

### なぜこの分離が有効か

- **再利用性**: Usecase内のVO検証は、HTTP経由/CLI経由/バッチ経由のどの入口からでも同じ保証を与える
- **テスト容易性**: Usecaseの単体テストで不正値を渡したときの挙動を直接検証できる
- **変更の局所性**: HTTPスキーマが変わってもVOは影響を受けず、ドメインルールが変わってもHTTPスキーマは影響を受けない
- **責務の明確化**: 「この検証は何を守っているか」がコードの配置から自明になる
