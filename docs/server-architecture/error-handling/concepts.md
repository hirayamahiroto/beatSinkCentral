# エラーハンドリング: メンタルモデルと設計原則

このドキュメントは、エラーハンドリング設計の **思想と原則** を扱う。「エラーとは何か」「なぜ層を分けるのか」「層の独立性とは何か」という問いに答える。

実際にどの層に何を置くかは [layer-responsibilities.md](./layer-responsibilities.md)、コードレベルの書き方は [implementation.md](./implementation.md) を参照。

---

## エラーとは何か（メンタルモデル）

> **エラーとは、ある処理の実行中に、その処理が依拠しているルール（契約・前提条件・不変条件）への違反が検知されたことを、呼び出し側に報告するための値である。**

「想定外のことが起きた時に返されるもの」という受動的な捉え方では解像度が低い。エラーは「実行された処理において、**何のルールが、どう違反されたか**」を能動的に表明するものだと捉える。

### 定義の要素分解

- **実行中に** — 静的に気づける問題（型エラー等）とは別。ランタイムで検知される事象を扱う
- **依拠しているルール** — 明示的に定義されている必要がある。ルールがなければ違反もない。だからレイヤーごとにルールを持ち、エラーもレイヤーごとに定義される
- **違反が検知された** — 誰かが検知している。検知責任はそのルールを知っている主体（＝そのレイヤー）が持つ
- **呼び出し側に報告** — 受け取る相手がいる値。だから情報を載せる価値があり、捨てるべきでもない

### 正常系とエラーは対等な結果

処理を実行したときに起こりうる結末は次の2つ。

- ルールがすべて守られた → 正常系の結果を返す
- どこかでルール違反が起きた → どのルールが破れたかを報告する（エラー）

どちらも「実行した結果」であって、片方が主役で片方が例外ではない。エラーは正常系の裏面であり、**同じ実行の別結果**。関数のシグネチャを `Input → Output` ではなく `Input → Output | Error` と捉えると、エラーも必ず処理すべき分岐のひとつになる。

### ルールはレイヤーごとに異なる

| レイヤー       | そのレイヤーが持つルール | 違反時のエラー例                           |
| -------------- | ------------------------ | ------------------------------------------ |
| Value Object   | 値の形式ルール           | emailは@を含む、subIdは空でない            |
| Entity         | 状態の整合性ルール       | 公開状態なら必須項目が埋まっている         |
| Usecase        | 業務ルール（前提条件）   | 新規登録は未登録ユーザーに対してのみ行える |
| Infrastructure | 依存先との契約ルール     | DBは応答する、外部APIは時間内に返す        |

ルールを持つレイヤーがそのルールへの違反（エラー）を定義する。ルールを知らないレイヤーはそのエラーを投げられないし、投げるべきでもない。これが「エラーは発生したレイヤーで定義する」という原則の根拠。

### このモデルから導かれる帰結

- エラー名は「違反されたルールの名前」になる（例: `UserAlreadyRegisteredError` = 「新規登録の前提である未登録状態」への違反）
- エラーには「何の違反か」という情報が型として載るべき
- 握りつぶし = 違反の報告書を破り捨てる行為。「誰が悪かったか」「どう直すか」「再発するか」が全て失われる

## なぜこの設計が必要か

一言でいうと「**エラーを情報として扱い、失敗の原因と対処を呼び出し側が正しく判断できるようにするため**」。

### 解決したい課題

#### 1. 「どのレイヤー由来のエラーか」が判別できない問題

エラーが全て素の `Error` で投げられていると、受け取った側は以下を区別できない。

- 値が不正（ユーザーの入力ミス → 400で返すべき）
- 業務ルール違反（既に登録済み → 409で返すべき）
- 外部システム障害（DB接続失敗 → 500で返すべき）

結果、全部500で返すか、メッセージ文字列で `includes("already")` のような脆い判定をする羽目になる。型（`type` フィールド + 型ガード関数）で判別できるようにすることで、呼び出し側が安全に分岐できる。

#### 2. エラーの発生原因と対処責任の場所がぼやける問題

「既に登録済み」をドメインエラーとして投げると、ドメイン層がユースケース固有の事情（新規登録フローでしか意味がない制約）を知っていることになる。別のユースケース（例: 管理画面からの強制再同期）では「既に登録済み」はエラーですらないかもしれない。

エラーは **発生したレイヤー＝そのルールを知っているレイヤー** で定義する。レイヤー責務とエラー定義を一致させることで、「どこを直せばこのエラーが変わるか」が明確になる。

### エラーを握りつぶすと何が起きるか

握りつぶし = `catch (e) {}` や `catch (e) { return null }`、あるいは全エラーを一律500にマッピングする実装のこと。次のような問題が起きる。

1. **ユーザーに正しいフィードバックが返せない**
   「既に登録済み」なのに「サーバーエラー」と表示されると、ユーザーはリトライを繰り返す。本来は別のアクション（ログイン導線への誘導）が必要。

2. **本物の障害が埋もれる**
   DB接続失敗も業務ルール違反も同じ扱いになるので、監視・アラートで「いま本当に危険なエラー」を拾えない。500が1日1000件出ていても、そのうち何件が本物の障害か誰もわからなくなる。

3. **リトライ戦略が壊れる**
   一時的な障害（リトライで回復する）と恒久的な失敗（何度やっても同じ結果）を区別できないと、クライアントは無駄なリトライを繰り返すか、リトライすれば成功するケースを諦めてしまう。

4. **デバッグ情報が失われる**
   `catch (e) {}` で握りつぶすとスタックトレースも原因も消える。数ヶ月後に似たバグが出たとき、過去に同じエラーが起きていた事実すら残っていない。

5. **セキュリティ的な混乱**
   「認可違反（403）」と「リソース不存在（404）」を区別しないと、攻撃者に「存在するが権限がない」情報を与えてしまうケースや、逆にユーザーに本来返すべき情報を隠してしまうケースが混ざる。

### 本質

- 握りつぶし = **「失敗した」という最低限の事実しか残らない状態**
- 責務分離したエラー定義 = **「どのレイヤーの、どのルールに、どういう文脈で違反したか」を型に載せて伝える状態**

呼び出し側は型を見て分岐できるので、次を正しく判断できる。

- ユーザーに何を伝えるか
- リトライすべきか
- ログレベルをどうするか
- アラートを上げるか

これが「エラーを情報として扱う」ことの意味。

## 基本原則

- エラーは **発生したレイヤーで定義** し、そのレイヤーの責務範囲内の事象を表現する
- 下位層は上位層を知らない。エラーは型として上位層へ伝播する
- エントリポイント層で各レイヤーのエラーを捕捉し、HTTPレスポンスへマッピングする

各層に何を置くかの具体は [layer-responsibilities.md](./layer-responsibilities.md) を参照。

## 層の独立性とは何か

エラーハンドリングもバリデーションも、根底にあるのは「層が独立している」という前提。この前提を言語化しておく。

### 各層は「異なる問いに答える専門家」である

「各層は責務が違う」という表現は正しいが、一段深くするとこう言える。

> **各層は、それぞれ固有の「問い」を持ち、他の層に代替不可能な答えを出す存在である。**

層ごとの問いを並べると次の通り。

| 層                     | 答えている問い                             | 失敗時の意味                         |
| ---------------------- | ------------------------------------------ | ------------------------------------ |
| **エントリポイント層** | このリクエストは受け取れる形で来ているか？ | 呼び出し側がプロトコルを守っていない |
| **Usecase層**          | この業務操作はいま実行してよいか？         | リクエストは正しいが業務状態的に不可 |
| **ドメイン層**         | この値・状態はビジネス概念として正当か？   | 値そのものが存在を許されない         |
| **Infrastructure層**   | 依存先は約束通り応答したか？               | 自分の責任ではない外部要因           |

これらの問いは**互いに独立**で、**1つを答えても他は答えたことにならない**。

### 誤解: 「層は同じ処理を段階的に厳密化している」

よくある誤った読み方:

> 「Zodで緩く見て、VOで厳しく見る。段階的に精緻化している」

これは間違い。正しい理解は次の通り。

> **ZodとVOは別々の事柄を別々にチェックしている。「非空」が両方に現れるのは、偶然の一致であって段階的精緻化ではない。**

- Zodが守っているのは **HTTPプロトコル上の約束**
- VOが守っているのは **ドメイン概念としての存在可能性**

これらは「同じ `email` フィールドに対する検証」に見えても、**守っている世界が違う**。Zodは「HTTPの世界」の守護者であり、VOは「ドメインの世界」の守護者。2つの世界が一瞬同じ文字列を経由するだけ。

### 責務とは「変更理由」である

単一責任原則の本質は「変更理由が1つであること」。この視点で層を再定義すると次のようになる。

| 層                 | 変更される理由                            |
| ------------------ | ----------------------------------------- |
| エントリポイント層 | HTTPプロトコル・APIスキーマが変わったとき |
| Usecase層          | 業務フロー・前提条件が変わったとき        |
| ドメイン層         | ドメインモデルのルールが変わったとき      |
| Infrastructure層   | 依存先のAPI・技術的実装が変わったとき     |

**変更理由が違う = 別のタイミングで変更される = 別の層にあるべき**。

「HTTPスキーマが変わっただけなのにドメイン層も直す必要がある」という状態は、責務が混ざっている証拠。逆に「ドメインルールが変わったのにHTTP層も直す必要がある」も同じ。

### 層は「知識の非対称性」を作る

クリーンアーキテクチャの核心として、層は**一方向の依存**を持つ。

- エントリポイント層はUsecaseを知っている
- Usecaseはドメインを知っている
- ドメインは何も知らない

この非対称性の意味はこうなる。

> **下位層（ドメイン）は、自分がHTTPから呼ばれていることを知らずに済むべき。そうあるからこそ、HTTP以外からも呼ばれ得る。**

ドメイン層が「自分はAPIリクエストの一部として呼ばれた」と知ってしまうと、CLIから呼んだときに挙動が変わってしまう。知らないからこそ、どこから呼ばれても同じ契約を守れる。

テストでも同じ。ドメインの単体テストはHTTPを知らずに書ける。HTTP層のテストはドメインの詳細を知らずに書ける。

### 層が「独立していること」の3つの意味

1. **問いが独立**: 各層は固有の問いに答える。他の層の問いを代行しない
2. **変更理由が独立**: 各層は別のタイミングで変更される。1箇所の変更が他に波及しない
3. **知識が独立**: 下位層は上位層を知らない。どこから呼ばれても同じ契約を守る

この3つが揃っているとき、層は「ただの整理のためのフォルダ分け」ではなく、**設計上意味のある境界**になる。逆にこの3つのどれかが崩れていると、層名だけがあって実態は混ざっている、という状態になる。

## ドメイン層が知っていること・知らないこと

エラー設計の責務境界を「ドメイン層が何を知って、何を知らないか」という切り口で言語化する。この線引きが守られていれば、メッセージ・HTTPステータス・レスポンス形式などのプレゼンテーション関心はすべて errorMap（翻訳層）に閉じられる。

### ドメインの知識範囲

| 項目                                       | ドメインは知る | ドメインは知らない |
| ------------------------------------------ | -------------- | ------------------ |
| どの条件でエラーになるか（ルール）         | ✓              |                    |
| どの型のエラーを投げるか（discriminator）  | ✓              |                    |
| 構造化コンテキスト（accountId, userId 等） | ✓              |                    |
| クライアントに見せる文言                   |                | ✓                  |
| HTTP ステータスコード                      |                | ✓                  |
| ログフォーマット                           |                | ✓                  |
| 国際化・多言語                             |                | ✓                  |
| レスポンスの JSON 構造                     |                | ✓                  |

ドメイン側にあるのは「何のルールが、どういう状況で破られたか」という **情報の発信者としての最小限の知識** だけ。その情報をどう整形して誰に届けるかは、ドメインの外側（errorMap / onError）の責任。

### コード上の対称性

```typescript
// domain 側: 何のルールに違反したか、状況の手がかりは何か、だけを表明
throw createTypedError("AccountIdAlreadyTakenError", { accountId });
```

```typescript
// errorMap 側: HTTP の作法と人間向け文言に翻訳
AccountIdAlreadyTakenError: {
  status: 409,
  message: (error) => `Account ID already taken: ${error.accountId}`,
},
```

ドメインの `throw` には HTTP ステータスも英語の文言も含まれない。errorMap はドメインの発信した「型 + context」を受け取って、プレゼンテーションの作法に翻訳する。

### この分離が効いてくる場面

#### 1. ドメインを別プロトコルから呼ぶとき

CLI / バッチ / Worker から同じ domain を使うとき、HTTP status や JSON レスポンスは意味がない。だが「どのルールが破られたか」は同じく必要。

ドメインが HTTP を知らないからこそ、**ドメインに1行も手を入れずに** 別プロトコルから使い回せる。

#### 2. メッセージ文言の変更

「Account ID already taken」→「アカウントIDは既に使用されています」という変更は、errorMap の1行だけ書き換えれば完結する。domain / policy / VO のコードは**触らない**。

文言変更が業務ルールの変更ではないのに、両方のレイヤーに書き換えが及ぶなら責務が混ざっている証拠。

#### 3. 同じエラーに対して異なるプレゼンテーション

同じ `AccountIdAlreadyTakenError` に対して:

- **API レスポンス**: `{ error: "Account ID already taken: xxx" }`（簡潔）
- **内部管理画面**: バリデーション詳細まで表示
- **Slack通知**: 絵文字付きで警告
- **監視アラート**: 構造化ログから集計

のように複数のプレゼンテーションをぶら下げられる。どれもドメインの外で分岐させればよく、ドメインはどれも知らなくていい。

#### 4. ログと API レスポンスの分離

クライアントには generic なメッセージだけ返し、サーバログには zod の issue detail を構造化して残す、といった「見せる粒度を相手ごとに変える」設計がドメイン側を変えずに可能になる。

### ドメインが「知っていい」ギリギリの線

構造化コンテキスト（例: `AccountIdAlreadyTakenError` の `accountId` フィールド）はドメインが知っていて **OK**。なぜなら:

- `accountId` は **ドメインの語彙**（ビジネスの用語）
- 「どのアカウントIDで衝突したか」という情報はドメインルール違反の**実体そのもの**

一方、ドメインが持ってはいけないのは:

- `"Account ID already taken: xxx"` というフレーズ（英語という選択、人間向け整形）
- `409` という数値（HTTP の作法）
- `{ error: "...", issues: [...] }` という構造（JSON レスポンス形式）

### チェック基準

ある情報をドメインに置いていいか迷ったときの判定:

- それは **業務用語で語れるか？** → Yes なら ドメイン OK
- それは **プロトコル・表示・運用の都合か？** → Yes なら ドメイン外に置く

この線引きが守られていれば、「ドメインが知る必要がないものまで抱え込んでいる」状態を避けられる。
