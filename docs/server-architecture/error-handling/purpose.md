# エラーハンドリング設計の目的と効くタイミング

このドキュメントは「型付きのエラー定義」「レイヤー別の責務分離」「クライアント向けと内部ログの分離」といったエラーハンドリング設計が、**実際にどこで・誰にとって価値を生むか** を整理したもの。設計思想そのものは [concepts.md](./concepts.md) を、層ごとの責務は [layer-responsibilities.md](./layer-responsibilities.md) を、実装パターンは [implementation.md](./implementation.md) を参照。

「なぜこのコストを払って設計するのか」を、実装者・運用者・新規参画者が共有するための材料として残す。

## 概要

エラーは「失敗した」という事実だけではなく、「何のルールが、どう違反されたか」を伝える情報資産である。型と構造化情報を伴って正しく扱われると、次の9つの異なる価値が生まれる。

```text
ユーザー体験 ─┐
監視・運用    │
障害分析      │
リトライ戦略  │── すべてが「同じエラー設計」から派生する
セキュリティ  │
ビジネス指標  │
プロトコル独立│
多言語化      │
開発体験      ─┘
```

逆に握りつぶし（catch して 500 に丸める）や文字列マッチ（`includes("already")`）で運用すると、これらの価値はすべて失われる。最初は「冗長なコスト」に見えるが、規模・運用期間が伸びるほど投資効果が顕在化する。

---

## 視点別の価値

### 1. ユーザー体験

**価値**: ユーザーに「次に何をすればいいか」が伝わる

| エラー種別                          | ユーザーが取るべき行動                             |
| ----------------------------------- | -------------------------------------------------- |
| `UserAlreadyRegisteredError`（409） | ログイン導線へ誘導。リトライしても無意味           |
| `AccountIdAlreadyTakenError`（409） | 別の名前を提案。フォーム下に inline 表示           |
| `InvalidEmailFormatError`（422）    | 入力修正。フィールドにフォーカス                   |
| 接続障害（500）                     | しばらく待ってリトライ。ユーザーの入力ミスではない |

全部を 500 で返してしまうと、ユーザーは「自分が悪いのか、システムが悪いのか」が判断できず、無駄なリトライや離脱、サポート問い合わせに繋がる。

### 2. 監視・アラート精度

**価値**: 「本物の障害」と「想定内の業務エラー」を分離して扱える

- **4xx（業務ルール違反）の急増** → UX 改善のシグナル（フォームのわかりにくさ、フローの分岐ミス等）
- **5xx の急増** → 即アラート、オンコール召集の対象
- 「重複登録 1日100件」が当たり前に出るサービスで、それを 500 として混ぜると **本物の障害が埋もれる**

これがエラー設計の最大の運用価値。SLO/SLI 計算でも、エラー率の指標が現実を反映する。

### 3. 障害分析（インシデント対応）

**価値**: 構造化情報で原因の当たりが速くつく

サーバーログの構造化フィールドにより、次のような切り口で素早く原因を特定できる:

- `type=AccountIdAlreadyTakenError accountId=xxx` でログ検索 → 「同じユーザーが何度も同じエラーを踏んでいる」を発見
- `type=*` で集計し時系列に並べる → 「特定時間帯に DB 接続失敗だけ急増」を可視化
- スタックトレース + 構造化コンテキストにより、後追い再現の精度が上がる

握りつぶしや文字列マッチに比べて、調査時間が桁違いに短縮される。

### 4. リトライ戦略

**価値**: クライアント側の自動リトライが「正しい時だけ」走る

| エラー性質                                 | リトライ可否                             |
| ------------------------------------------ | ---------------------------------------- |
| 一過性(接続失敗、タイムアウト、503)        | exponential backoff でリトライ妥当       |
| 恒久的失敗(バリデーション、業務ルール違反) | 何度試しても結果は同じ。**リトライ禁止** |

HTTP status と `code` で機械可読に区別できれば、SDK / クライアントが自動判定できる。区別できないと、ユーザーが無意味なリトライを繰り返したり、復旧可能なエラーを諦めたりする。

### 5. セキュリティ（情報漏洩防止）

**価値**: 攻撃者にヒントを与えない、ユーザー列挙攻撃を防げる

- 「存在するが権限がない」と「存在しない」を区別するか・隠すかを **設計で意識的にコントロール** できる
- SQL クエリ文字列、テーブル名、`constraint_name`、スタックトレースが **クライアントに漏れない**（`message` のみ返す方針で達成）
- エラーメッセージ経由のユーザー列挙攻撃（「このメールは登録済みです」と「未登録です」の出し分けで攻撃者に情報を与える）を、文言設計で抑止できる

### 6. ビジネスメトリクス

**価値**: エラー自体が事業上の指標になる

- `AccountIdAlreadyTaken` の頻度 → 「人気のアカウント名」「ユーザーが思いつきやすい名前」の傾向が見える
- `UserAlreadyRegistered` → リテンション分析の材料（再訪ユーザー数の代理指標）
- バリデーションエラー（422）の項目別頻度 → フォーム改善の優先順位付け
- ファネル分析で「どこで脱落したか」が **エラー種別から定量的に見える**

「エラーは隠すもの」ではなく「ユーザー行動を映す鏡」として活用できる。

### 7. プロトコル独立性

**価値**: 将来の構造変更コストが下がる

ドメイン層が HTTP を知らないため、CLI / バッチ / Worker / GraphQL / gRPC など **任意の入口から同じユースケースを呼べる**。

具体例:

- データ移行スクリプトでも `assertAccountIdAvailable` の重複チェックロジックが効く
- 管理画面用の Worker から同じ業務ルールを呼び出せる
- 将来 API のプロトコルを GraphQL に乗せ換えても、entrypoint だけ書き換えればドメインは無傷

入口を増やしても、業務ルール本体は1箇所のまま。

### 8. 多言語化・マルチクライアント

**価値**: 拡張時の改修範囲が局所化される

- `errorMap` 1ファイルを言語別に切り替えるだけで多言語対応できる
- mobile / desktop クライアントを足した時、`code` で UI 挙動を出し分けられる余地（Issue #135 で議論した拡張ポイント）
- 「文言だけサーバー集中管理 / クライアントで翻訳」を後から切り替えても、ドメイン層は無傷

### 9. 開発体験 / コード品質

**価値**: 新しい機能を追加する時に「考えるべきこと」が型で誘導される

- 新しいユースケースを書くとき、「どのレイヤーのどんなルールに違反するか」を分類する習慣が型システムから要請される
- `errorMap` の `Record<AppError["type"], ...>` で **未対応エラーがコンパイル時に検出できる**
- テストが書きやすい: assertion は純粋関数（DB / HTTP 不要）、エラーパスも分岐ごとに単体テストできる
- 新人が入った時、**エラー型の一覧そのものが業務ルールのドキュメント** として機能する

## いつ効くか（タイムライン）

設計コストの回収タイミングは段階的。逆に「後から入れる」のは既存エンドポイントの全面改修になりコストが跳ねるため、最初の段階で型で固めておく価値が大きい。

| 効く時期                         | 主に効く価値                                       |
| -------------------------------- | -------------------------------------------------- |
| **今すぐ（最初のユーザーから）** | ユーザー体験 / セキュリティ                        |
| **数十ユーザー〜運用開始**       | 監視精度 / 障害分析                                |
| **スケール期**                   | リトライ戦略 / ビジネスメトリクス                  |
| **複雑度・チーム規模が増した時** | プロトコル独立性 / 多言語化 / 新人オンボーディング |

実装初期は「冗長なコストに見える」が、運用が回り始めると **「投資しておいてよかった」と最も実感されるレイヤー** がエラー設計。

## 反対側のシナリオ: この設計をしないと何が起きるか

実害をイメージしやすくするため、設計を放棄した場合に何が起きるかを並べておく。

| 怠った場合                                 | 起きること                                                             |
| ------------------------------------------ | ---------------------------------------------------------------------- |
| 全エラーを 500 に丸める                    | ユーザーは無駄リトライ。本物の障害が埋もれる。SLO は意味を失う         |
| 文字列マッチで分岐                         | DB エンジンを変えた瞬間に判定壊れる。テストもメッセージ依存で壊れる    |
| エラーをドメインに持ち込まずインフラで処理 | CLI / バッチから同じロジックを呼べない。レイヤー越境で変更影響が広がる |
| クライアントに stack や SQL を返す         | 情報漏洩、ユーザー列挙攻撃の踏み台                                     |
| 握りつぶし（`catch (e) {}`）               | 障害が静かに発生。誰も気付かないまま被害が拡大                         |

これらの「怠ったらどうなるか」が、設計の価値を裏側から照らす。
