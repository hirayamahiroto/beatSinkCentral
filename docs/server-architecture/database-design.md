# DB設計指針

## このプロジェクトにおけるテーブル設計の共通原則

---

## 設計の軸

```
DBの責務　→　信頼できる事実を記録する
アプリケーションの責務　→　振る舞いの実行と判断
```

新しいテーブルを設計するとき、すべての判断はこの軸に戻って考える。

---

## 原則

### 1. ドメインを定義してから設計する

テーブル構成より先に「これは何か」を言語化する。定義が曖昧なまま設計を始めると、カラムの置き場所で迷子になる。

```
// 悪い例：設計から入る
「UserとArtistをどう繋げるか」→ とりあえずFKを持たせる

// 良い例：定義から入る
「Artistは表現者である」→ soloもteamも同じテーブルでよい
→ 関係の多重度を確認 → 中間テーブルの必要性を判断
```

### 2. typeカラム・フラグカラムを使わない

DBに振る舞いの判断を持たせることになるため採用しない。

```
// 悪い例
type: "solo" | "team"     → DBが分類の判断を持つ
is_active: true | false    → DBが状態の判断を持つ

// 良い例
artist_membersにレコードがある → teamとして機能する（構造が語る）
accepted_atがNULL           → 招待中（タイムスタンプの有無が語る）
```

### 3. 関係はテーブルで表現する

エンティティの属性と関係の情報を混在させない。

```
// 悪い例：関係をカラムに埋め込む
artists.user_id FK → users.id
→ 1:Nに拡張するときテーブル構造の変更が必要

// 良い例：関係を中間テーブルで表現する
artist_owners(user_id, artist_id)
→ 多重度の変化にテーブル追加で対応できる
```

### 4. 関係の多重度はドメインの実態で判断する

将来1:Nになり得るなら中間テーブルを選ぶ。FKから中間テーブルへの移行コストは高い。判断の根拠はドメインの実態に置く。

```
// 判断の問い
「このエンティティは将来複数の相手に紐づき得るか」
→ Yes → 中間テーブル
→ No（かつ将来もない） → FK
```

### 5. 今やらないことを明示する

将来の拡張方向性が見えていれば、今はシンプルに保つ。ただし「考えていない」と「意識的にやらない」は別物。設計ドキュメントに明示しておく。

---

## テーブル設計チェックリスト

新しいテーブルを追加するとき、以下を確認する。

```
□ このテーブルのエンティティを一言で定義できるか
□ 各カラムはそのエンティティの属性として自然か
□ 「関係の情報」が混入していないか
□ typeやflagカラムを使っていないか
□ タイムスタンプの有無で状態を表現できないか
□ 将来の多重度の変化を考慮したか
□ 今やらないことを明示したか
```
