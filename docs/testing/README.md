# テスト戦略 整備プラン

本ドキュメントは、テスト設計・実装・レビューをチーム共通の指針に乗せるための整備計画。ドラフト段階であり、Discussion / PR を通じて合意してから本実装に進む。

---

## 背景

現状、プロジェクト内でテストに関するドキュメントは「実行コマンド」までで、**テスト設計（何を・どの粒度で・どのレイヤーでテストするか）の方針** は文書化されていない。

### 無いもの

- テスト戦略（テストピラミッド、各レイヤーで何をテストするかの基準）
- テスト設計テンプレート・サンプル
- カバレッジ方針
- モック方針（DB・外部APIの扱い）
- レビュー時のテスト要件（実装PRに対して何が満たされていればOKか）
- UI / 機能変更時に推奨されるテスト粒度の判断基準

`docs/templates/` ディレクトリは存在しないが、テストデータテンプレート等が将来追加される予定（memory 記録あり）。

## 目的

テスト設計の判断・実装・レビューが、属人的・場当たり的にならず、**チーム共通の指針** に沿って行えるようにする。

---

## 整備フェーズ

本整備は一度に全てをやらず、**優先度に沿って段階的に積み上げる**。下記フェーズ順に進め、前フェーズが固まる前に次に進まない。

### Phase 1（現在）: Unit Test の整備

- 対象: Value Object / Entity / Policy / Usecase / Repository / Entrypoint の単体テスト（Sociable Unit Test 含む）
- 成果物: `strategy.md` / `guidelines.md` / `test-cases.md` / code-review-checklist 拡張 / 導線整備
- 理由: **最小単位から組み上げる実装方針の土台**。Unit が甘い状態で上位を積んでも意味を成さない
- 本プロジェクトのテスト戦略の **主軸** がここ

### Phase 2（後続）: Integration Test の整備

- 対象: Repository を実 PostgreSQL に対して検証する統合テスト
- 必要になるもの: Docker Compose の DB / Transaction rollback / fixture 機構 / CI 統合
- 理由: Drizzle クエリと実スキーマの整合、マイグレーション適用後の動作など、単体では検証不能な領域をカバー
- **Phase 1 が固まり、実装が増えて実 DB 乖離のリスクが具体化してから着手**

### Phase 3（必要なら）: E2E Test の整備

- 対象: クリティカルパス（例: 認証→登録→取得）の HTTP → 実 DB 一気通貫
- 理由: 原則として **Phase 1（Entrypoint 単体テスト）+ Phase 2（Repository 統合）で代替可能** なため必須ではない
- ただし本番デプロイ後のスモークテストは別途検討
- 設計候補は `e2e-strategy.md` に記録済み（今は読まなくていい）

### フェーズ跨ぎの判断基準

| 次フェーズへ進む条件                       | 指標                                        |
| ------------------------------------------ | ------------------------------------------- |
| Phase 1 → Phase 2                          | Unit Test のレビュー観点がチームで定着し、新規 PR で自然に守られている |
| Phase 2 → Phase 3                          | 実 DB 統合の恩恵が出てもなお拾えない不具合が発生・予見される |

## 各アーティファクトの想定目次

### 1. `docs/testing/strategy.md`

```text
1. 背景と目的
2. テストのメンタルモデル
   - テストは「仕様を実行可能な形で書いたもの」
   - 「壊れにくいテスト」の条件（実装詳細ではなくIOを検証）
3. テスト粒度の分類（テストピラミッド）
   - 単体 / 統合 / E2E の定義と比率指針
4. 層ごとのテスト責務（CLAUDE.md/architecture.md を拡張）
   - Value Object / Entity / Factory / Domain Service / Policy
   - Usecase / Repository / Entrypoint / UI
5. モック方針
   - 何をモックする / しない
   - DB / 外部API / 時間 / 乱数
6. カバレッジ方針
   - 数値目標ではなく「カバーすべき振る舞い」で管理
7. テスト命名・配置規約
   - {FuncName}/index.test.ts
   - describe/it の日本語表記ルール
8. エラーハンドリングのテストパターン（今回の設計を反映）
   - typed error の throw 検証
   - errorMap の変換テーブルテスト
9. 時間・乱数・環境変数の扱い
10. テストをいつ書くか（TDD の立ち位置）
```

### 2. `docs/testing/guidelines.md`

```text
1. Value Object のテスト（バリデーション網羅例）
2. Entity のテスト（IO 確認 + spy 例）
3. Policy / Domain Service のテスト（純粋関数の分岐網羅例）
4. Usecase のテスト（Repository モック例）
5. Repository のテスト（DB モック例）
6. Entrypoint（API ハンドラ）のテスト（Container モック例）
7. UI / Component のテスト（機能 / スナップショット / アクセシビリティ）
8. エラーハンドリングのテスト
   - factory → typed error
   - errorMap → status/message 変換
9. アンチパターン集
   - テストのために実装を改変している
   - モックの検証でテストが実装に張り付いている
   - 時間・乱数に依存して間欠的に落ちる
```

### 3. `docs/templates/test-cases.md`

```text
## テストケース設計チェックリスト

### 正常系
- [ ] ハッピーパス
- [ ] 典型的な入力範囲

### 異常系
- [ ] 入力の型不正 / 必須漏れ
- [ ] ビジネスルール違反
- [ ] 認可違反

### 境界値
- [ ] 最小 / 最大
- [ ] 空 / null / undefined

### 権限
- [ ] 各ロールでの挙動
- [ ] 他テナントのリソースへのアクセス拒否

### UI（該当する場合）
- [ ] レスポンシブ（SP/Tablet/PC）
- [ ] キーボード操作
- [ ] スクリーンリーダー
- [ ] ローディング / エラー状態の表示

### 監視観点（新規APIの場合）
- [ ] 想定されるエラー型が errorMap に登録されている
- [ ] 高頻度 4xx はログレベル info で出力される想定か
```

### 4. `.claude/rules/code-review-checklist.md` への追記

```text
## 14. テスト要件

- 新しい公開関数・公開APIにはテストが1件以上あるか
- バグ修正 PR には回帰テスト（= そのバグを再現するテスト）があるか
- 層を跨ぐ変更の場合、各層のテストが独立して通るか
- モックが「実装の内部呼び出し」ではなく「境界の契約」をモックしているか
- エラーハンドリング変更時:
  - typed error が throw されることをドメイン側でテスト
  - HTTP への変換は errorMap 側でテスト（二重に書かない）
```

### 5. 導線整備

- `CLAUDE.md` の「テスト方針」節から `docs/testing/strategy.md` / `guidelines.md` へリンク
- `OVERVIEW.md` の目次（あれば）にも追加

## 進め方

1. **[1] strategy.md の初版をドラフト** — Discussion / PR の叩き台として
   - チーム合意が取れるまでは他を止める（方針が固まらないと下流が書けない）
2. 合意後 **[2] guidelines.md** を書く（具体例が多いのでストラテジと分離）
3. 同時並行で **[3] test-cases.md テンプレート** を最低限で作る
4. **[4] code-review-checklist.md の「14. テスト要件」** 追加
5. **[5] 入口導線** を最後に整える
