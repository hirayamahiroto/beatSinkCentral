# E2E テスト戦略（設計候補）

本ドキュメントは E2E テスト方針を確定させるための **設計候補・論点一覧** である。軸ごとに選択肢と推奨を提示し、合意が取れた段階でこのドキュメント自体を確定版の戦略書に改訂する。

関連: `docs/testing/strategy.md`（プロジェクト全体のテスト戦略）

---

## ステータス

**Phase 3（未着手 / 現時点では読む必要なし）**。

本プロジェクトの整備フェーズは `docs/testing/README.md` を参照。現在は **Phase 1（Unit Test の整備）** を進めており、E2E の議論は先送りされている。

### なぜ先送りか

- Entrypoint の Sociable Unit Test で HTTP 層の正常動作は大部分カバー可能
- Repository の実 DB 統合テスト（Phase 2）で DB 整合は担保可能
- この2つが揃えば、API E2E の固有価値の多くは代替できる

### いつ再開するか

Phase 2 を終えてもなお以下のリスクが現実化した場合、E2E の導入を検討する:

- 層の繋ぎ目で拾えないバグが複数回発生
- デプロイ直後の障害が増加傾向にある
- ステージング環境でのスモークテストが追いつかない

それまで本ドキュメントは **将来の判断材料** として保管しておくのみ。以下の設計候補一覧はそのための記録。

## スコープ

`docs/testing/strategy.md` で定義した「積み上げ型テスト」の最上位レイヤー =「システム全体の組み上がった振る舞い」をカバーするテスト設計を対象とする。

- 単体テスト・統合テストは `strategy.md` / `guidelines.md`（将来）で定義
- 本書は「E2E テスト」としてどこからどこまでを検証するか、どの技術で実行するかを決める

## 決めるべき軸

### 軸A: テストの対象範囲

| 選択肢  | 説明                                              | 使い分け                       |
| ------- | ------------------------------------------------- | ------------------------------ |
| UI E2E  | ブラウザ操作（Playwright 等）で画面から DB まで   | ユーザー視点の全体確認         |
| API E2E | HTTP → Hono → 実 DB。ブラウザは介さない           | バックエンドの統合確認         |
| 両方    | 上記を併用                                        | フルスタックを堅く守りたい場合 |

**推奨**: 初期は **API E2E を主、UI E2E を後追い**。API E2E で「バックエンド全層の統合」を確認し、UI は Storybook + 手動 QA で補完 → フェーズ進行に応じて UI E2E を追加。

---

### 軸B: 使用ツール

| 選択肢                    | 向き              |
| ------------------------- | ----------------- |
| Playwright                | UI E2E 定番       |
| Vitest + Hono test client | API E2E に最適    |
| supertest                 | Express系には定番 |
| Cypress                   | UI E2E 代替       |

**推奨**: Hono を使っているので **`app.request()`（Hono の組み込みテスト機構）+ Vitest** が最速・最小依存。UI は将来 **Playwright** を追加。

---

### 軸C: 認証の扱い

| 選択肢                                 | 説明                                        |
| -------------------------------------- | ------------------------------------------- |
| Auth0 ステージングテナントを実際に叩く | リアル。ただし遅い・不安定                  |
| テスト用 JWT を直接発行                | 速い。Auth0 の検証ロジックはバイパス        |
| 認証ミドルウェアをテスト時に差し替え   | 完全にバイパス。auth の実装は別途単体で検証 |

**推奨**: **認証ミドルウェアをテスト時に差し替え**。E2E の目的は「認証後のフロー検証」であり、認証自体は別テストで見る。

---

### 軸D: データ戦略

| 選択肢                | 説明                                                     |
| --------------------- | -------------------------------------------------------- |
| Transaction rollback  | テスト毎に BEGIN → 実行 → ROLLBACK。他テストに影響しない |
| Truncate between tests | 各テスト開始時に全テーブル TRUNCATE                      |
| Ephemeral DB per test | 毎回 DB を新設（遅い）                                   |
| Seeded DB（参照のみ） | 読み取り系テストで固定データを使う                       |

**推奨**: **Transaction rollback**。速度と独立性を両立。読み取り系は seeded fixture を併用。

---

### 軸E: DB 環境

| 選択肢                       | 説明                                     |
| ---------------------------- | ---------------------------------------- |
| Docker Compose の PostgreSQL | ローカル・CI 共通で再現性◎               |
| `pg-mem` 等のインメモリ DB    | 速いが Drizzle や再帰 CTE との相性要確認 |
| テスト用リモート DB          | 保守コスト高                             |

**推奨**: **Docker Compose + PostgreSQL**。本番と同じ DB を使うのが最小驚き。

---

### 軸F: CI 実行頻度

| 選択肢                 | 説明                     |
| ---------------------- | ------------------------ |
| PR 毎に全 E2E          | 最高品質、最長 CI        |
| PR 毎に API E2E のみ   | 速度と品質のバランス     |
| main マージ時のみ      | PR CI を軽く保つ         |
| Nightly                | 時間のかかる UI E2E 向き |

**推奨**: **PR 毎に API E2E、UI E2E は main 時のみ**（初期）。

---

### 軸G: カバー範囲

| 選択肢                           | 説明                               |
| -------------------------------- | ---------------------------------- |
| 全ユーザーストーリーを網羅       | 工数大、保守負担大                 |
| クリティカルパス（主要導線）のみ | 最低限、壊れたら致命的なパスを守る |
| スモーク（起動確認レベル）       | 最小限                             |

**推奨**: **クリティカルパス主義**。「壊れたらリリースできない」導線だけ E2E で守る。エッジケースは単体/統合で。

---

### 軸H: 速度 SLO

| 選択肢   | 感覚                            |
| -------- | ------------------------------- |
| 1 分以内 | 理想。並列で分割すれば現実的    |
| 3 分以内 | API E2E の現実的ライン          |
| 5 分以上 | CI の体験が悪化しはじめる       |

**推奨**: **API E2E 3 分以内 / UI E2E 含めて 10 分以内**。

---

## 推奨方針のまとめ（全部 推奨に倣った場合）

```text
対象範囲:  API E2E 主（初期）、UI E2E は後追い
ツール:    Hono app.request() + Vitest（API）/ Playwright（UI 将来）
認証:      テスト時に認証ミドルウェアをモックに差し替え
データ:    PostgreSQL + Transaction rollback
環境:      Docker Compose
CI:        PR 毎に API E2E、main 時に UI E2E
範囲:      クリティカルパスのみ
速度:      API 3 分 / UI 含め 10 分
```

## 追加で決めたい観点

### 配置と命名

- **テストファイルの配置**
  - 案1: `apps/api-server/src/app/api/.../e2e/` （機能と同居）
  - 案2: `apps/api-server/tests/e2e/` （専用ディレクトリ）
- **現在の単体テストとの区別**
  - ファイル名規約（`*.e2e.test.ts` 等）を設けるか
  - vitest の `include` / `exclude` で分離するか

### fixture 管理

- テストデータ生成のユーティリティの配置
  - 案1: `tests/fixtures/` 直下
  - 案2: feature 単位で co-located (`users/fixtures.ts`)

### 並列実行

- DB アクセスが並列で衝突しないか（Transaction rollback なら原則 OK だが、セッション分離の確認が必要）
- 並列度の上限を CI で制御するか

### 環境変数

- E2E 用の `.env.test` を用意するか
- Docker Compose の接続情報を CI / ローカルでどう共有するか

### 認証モックの実装方式

- ミドルウェアを差し替える仕組み（DI / 環境変数 / テスト用エントリポイント）
- テスト中にユーザーを「誰として」リクエストするかの指定方法（ヘッダ / ヘルパー関数）

## 進め方

1. 上記の軸 A〜H で各方針を合意（推奨のまま or 変更）
2. 追加観点（配置・命名・fixture・並列・env・認証モック）も合意
3. 合意をもとに本ドキュメントを確定版として書き直す
4. 小さなクリティカルパス1本（例: `POST /users → getMe`）を試作して検証
5. 試作でつまづいた観点があれば方針に反映して改訂
