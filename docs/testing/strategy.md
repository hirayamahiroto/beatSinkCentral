# テスト戦略

本ドキュメントは、このプロジェクトにおけるテスト設計・実装・レビューの共通指針を定める。数値目標や網羅ルールではなく、**「何のためにテストを書くのか」「どの単位で何を検証するのか」** を言語化することを目的とする。

本プロジェクトは DDD + クリーンアーキテクチャを採用しており、テスト戦略もその構造に沿って設計する。

整備の段階的な進め方は `docs/testing/README.md` の「整備フェーズ」を参照。本ドキュメントは **Phase 1(Unit Test の整備)** 期の主指針にあたる。

---

## 0. 本プロジェクトにおけるテスト分類と優先度

一般的な分類とこのプロジェクトの立ち位置を先に明示する。整備の優先度はここから派生する。

### 一般的な分類

| 分類               | 特徴                                                                         |
| ------------------ | ---------------------------------------------------------------------------- |
| Solitary Unit Test | 対象関数だけ検証。collaborator は全てモック                                  |
| Sociable Unit Test | 対象＋同一モジュールの純粋な collaborator を実物で使う。実インフラは含まない |
| Integration Test   | **実インフラ(DB / HTTP / FS 等)を含む** 複数モジュール統合テスト             |
| E2E / System Test  | 境界から境界まで全部実物                                                     |

**Integration の必要条件は「実インフラを含む」こと**。モックで埋まっていればどれだけ多くのモジュールを跨いでも分類上は Unit Test。

### 本プロジェクトの現状と立ち位置

| レイヤー         | 現状のテスト                | 分類                | 備考                           |
| ---------------- | --------------------------- | ------------------- | ------------------------------ |
| Value Object     | 純粋関数の単体              | Solitary Unit       |                                |
| Entity           | Factory の IO + spy         | Sociable Unit       |                                |
| Policy           | 判定ロジックの単体          | Solitary Unit       |                                |
| Usecase          | Repository をモック         | Sociable Unit       | 実 DB は含まない               |
| Repository       | DB をモック                 | Sociable Unit       | 実 DB は含まない               |
| Entrypoint (API) | Hono `app.request()` で実行 | Sociable Unit       | DB / Auth0 はモック            |
| Repository(統合) | —                           | Integration(未整備) | Phase 2 で追加予定             |
| E2E              | —                           | —(予定なし)         | Phase 3 の候補。原則不要と判断 |

つまり **現状は全て Unit Test**(Solitary / Sociable)で構成されており、Integration と E2E は未整備。

### テストピラミッドとしての位置づけ

本プロジェクトは **テストピラミッド型** を採用する。

```text
         /\
        /  \    E2E         ← 原則整備しない(必要ならクリティカルパスのみ)
       /----\
      /      \   Integration ← Phase 2 で Repository のみ最小限導入
     /--------\
    /          \  Unit        ← Phase 1 の主軸。厚く整備する
   /------------\
```

基本思想:

> **Unit Test で「ロジックとして期待通り動く」ことを担保し、**
> **Integration Test で「実環境と繋げても動く」ことを担保する。**

この役割分担を守ることが、ピラミッドが壊れずに保守可能であり続ける条件である。Integration でビジネスロジックの分岐網羅を試みたり、E2E で境界値を確認したりすると、ピラミッドが逆転して遅く壊れやすいテスト群になる。

### 優先度の方針

本プロジェクトの整備方針は以下の優先順:

1. **Unit Test を徹底する(主軸 / Phase 1)**
   - 積み上げ型実装の土台。ここが甘いと上位層の検証が意味を成さない
   - Entrypoint の Sociable Unit Test は「HTTP 層の正常動作」を単体テストの形で担保する
2. **Integration Test は Phase 2 で最小限導入**
   - Repository の実 DB 統合テストのみ導入
   - Drizzle ↔ 実 PostgreSQL の整合・マイグレーション適用後の動作を担保
   - 目的は **Unit でモックしている契約と、実装の契約の一致を保証する** こと(詳細は セクション 7)
3. **E2E Test は原則整備しない**
   - Entrypoint Sociable Unit + Repository Integration で代替可能
   - 必要になったらクリティカルパス 1〜3 本のみ追加

この優先順は「**テストは Unit から積み上がる**」という積み上げ型の設計方針と整合する。Integration / E2E を先行整備しても、Unit が甘ければ信頼性を生まない。

## 1. テストの目的

> **実装の内部がどう変わっても、期待している I/O が担保されていれば、全体を組み上げたときに期待した動作になる。**
> テストはその「I/O 担保」を自動化された証拠として残す仕組みである。

### 核心

- テストが守るのは **関数の入出力契約**
- テストが守らないのは **内部実装の詳細**(分岐の具体的な書き方、private な一時変数、呼び出し回数など)

この区別を持つことで次が得られる:

- **内部実装を自由にリファクタできる**(テストが壊れない限り振る舞いは変わっていない)
- **バグの検出と修正の境界が明確になる**(テストが通らない = I/O 契約違反)
- **変更の影響範囲が I/O で定義される**(契約が変わらなければ呼び出し側への波及はない)

### テスト = 実行可能な仕様

テストは「コードの追加コスト」ではなく「**仕様を実行可能な形で書いたもの**」として扱う。

- コメントで仕様を書いても腐るが、テストは腐らない(腐ったら赤になる)
- ドキュメントの表現と実装のズレはテストが暴く
- 新規参加者は「どう動くか」をテストから読む

## 2. 前提条件: 疎結合な設計

テスト戦略は **設計の疎結合性** に依存する。依存が絡み合っていれば、どう頑張ってもテストは書きにくい・壊れやすい・意味を失う。

### この戦略が機能する前提

- 各レイヤーが **意味のある単一の単位** である(Value Object は値、Policy は判定、Usecase は業務フロー、等)
- レイヤー間の依存が **明示的な contract**(インターフェース / 引数 / 戻り値の型)で繋がっている
- 下位層は上位層を知らない(依存方向が単方向)
- 副作用は境界(Repository / API クライアント)に集約され、中核は純粋関数

### テストが書きにくいのは設計の警告

テストを書いてみて次のような症状が出たら、**テストの問題ではなく設計の問題** である可能性が高い:

- 1つの関数をテストするのに大量のモック積み上げが必要
- テストがコードの内部呼び出し回数を検証せざるを得ない
- 無関係な機能を変えただけで大量のテストが落ちる

こうなったら **設計に戻って疎結合化する**。テストの工夫で取り繕わない。

## 3. 積み上げ型テスト(Atomic Design 類推)

実装方針として「**最小単位の実装を組み上げていく**」アプローチを取る。テストも同じ積み上げ方で設計する。

### 積み上げの構造

```text
[Value Object / 純粋関数]   ← 最小単位。単独で I/O を固める
         ↓ 組み合わせる
[Entity / Aggregate]        ← 最小単位同士の協調の振る舞いを固める
         ↓ 組み合わせる
[Policy / Domain Service]   ← ドメインルール・協調を固める
         ↓ 組み合わせる
[Usecase]                   ← 業務フロー・前提条件の振る舞いを固める
         ↓ 組み合わせる
[Repository / Entrypoint]   ← 外界との境界変換を固める
         ↓
[システム全体]              ← E2E で組み上げた結果を確認(原則不要)
```

### 積み上げの利点

- **下層が信頼できるから、上層で下層の詳細を再検証する必要がない**
- Value Object のバリデーションは VO 側で固める → Entity のテストでは VO の詳細を再検証しない
- Usecase は Repository をモックして業務フローだけ検証 → DB の詳細は Repository の Integration Test で固める(Phase 2)
- 積み上げ構造が壊れている(下層で甘い検証、上層で重複検証)と、テストはすぐに保守不能になる

### レイヤーごとに「振る舞いの質」が違う

重要な認識: **単体の実装と、それを組み上げたレイヤーの実装では、責務が違うので振る舞いも違う**。したがって検証対象も違う。

- **Value Object の振る舞い**: 「この値は正当か」を判定する
- **Entity / Aggregate の振る舞い**: 「複数の値・関連が整合しているか」を保持する(不変条件)
- **Usecase の振る舞い**: 「ユーザーストーリーを完了できるか」を実行する
- **Entrypoint の振る舞い**: 「プロトコル境界を超えて正しく通信できるか」を成立させる

各レイヤーで検証する振る舞いが違うため、テストの **設問が違う**。同じ設問を使い回すと、どこかの層でテストが冗長になるか、どこかの層で責務が抜け落ちる。

## 4. レイヤーごとのテスト責務

積み上げ構造に沿って、各レイヤーで「何を問うテストを書くか」を定義する。

| レイヤー                   | 検証する振る舞い                        | 典型的なテスト対象                                                                      |
| -------------------------- | --------------------------------------- | --------------------------------------------------------------------------------------- |
| **Value Object**           | 値の正当性                              | 有効な値で生成できる / 無効な値で正しく型付きエラーを投げる / 境界値                    |
| **Entity**                 | 生成時の IO + 状態の整合性              | 有効な入力で Entity が構築される / 不変条件違反でエラー / ID などの自動生成が期待通り   |
| **Aggregate (集約ルート)** | 集約内の不変条件 + ドメインイベント発火 | 集約ルート経由での状態変更 / 不変条件違反でエラー / 期待したドメインイベントの発火      |
| **Factory**                | 生成の契約                              | 入力から期待する Entity/VO が作られる                                                   |
| **Domain Service**         | 複数オブジェクトの協調                  | 組み合わせロジックが正しく動く / 前提条件違反で型付きエラー                             |
| **Policy (assertXxx)**     | ルール判定                              | 正常時は通過 / ルール違反時は期待する型のエラーを throw                                 |
| **Usecase**                | 業務フロー・分岐                        | 前提を満たすと業務が完了する / 前提違反で型付きエラー / 例外時のリソースロールバック    |
| **Repository**             | 永続化境界の契約                        | 保存・取得・更新・削除の I/O / DB エラーの伝播(**実装は Integration Test で検証**)      |
| **Entrypoint (API)**       | プロトコル変換                          | リクエスト → Usecase 呼び出し → レスポンス / 認証・認可 / errorMap を通じた HTTP 変換   |
| **UI / Component**         | ユーザー操作に対する表示・反応          | 初期描画 / ユーザー操作での状態変化 / ローディング・エラー状態の表示 / アクセシビリティ |

### 原則

- **各レイヤーでそのレイヤーが持つ振る舞いだけを検証する**
- 下層の詳細を上層で再検証しない(例: Usecase のテストで email の形式を検証しない。それは VO 側で済ませる)
- 上層の都合で下層のテストを変えない(Usecase で使われ方が変わっても、VO のテストは影響を受けない)

### 集約ルート / ドメインイベントの扱い

- 集約内の子 Entity は **集約ルート経由でのみ** 変更されるという不変条件を検証する
- 集約ルートのテストでは **発火したドメインイベントの種類と内容も検証対象** とする(副作用ではなく契約として扱う)
- イベントハンドラ側のテストはハンドラ単独で書く(イベント発火側とは独立)

## 5. モック方針

### 基本スタンス

> **テストデータは基本的にモックを作成する。**

モック生成は各テストファイルで完結させず、再利用しやすい形でヘルパー化することを推奨(`<feature>/fixtures.ts` など)。

### モックする対象 / しない対象

| 対象                    | モックするか | 理由                                    |
| ----------------------- | ------------ | --------------------------------------- |
| Value Object            | しない       | 純粋。素で呼べば済む                    |
| Entity / Aggregate      | しない       | 純粋。ファクトリで生成できる            |
| Policy / Domain Service | しない       | 純粋関数。引数を渡すだけ                |
| Repository              | する         | DB 依存を境界で遮断したい               |
| 外部 API クライアント   | する         | ネットワーク依存を境界で遮断したい      |
| 時間 (Date, Clock)      | する         | 決定的なテストのため `vi.useFakeTimers` |
| 乱数 / UUID             | する         | 決定的なテストのため固定化              |
| 環境変数                | する         | テスト実行時の環境に依存させない        |

### 「契約をモックする」原則

モックは **境界の契約** をモックする。**実装の内部呼び出し** をモックしない。

```typescript
// ✅ 良い: Repository の契約(findBySub というメソッドがあり、User | null を返す)をモック
const userRepository = { findBySub: vi.fn().mockResolvedValue(null) };

// ❌ 悪い: 内部で使っている SQL クエリ関数をモック
vi.spyOn(internalSqlBuilder, "select").mockReturnValue(...);
```

前者はリファクタに強い(Repository が内部でどう動いても壊れない)。後者は実装を変えると必ず壊れる。

### モックの「嘘」への対処

Unit Test におけるモックは **書いた人が信じている契約** であって、実装の契約そのものではない。モックと実装が乖離していた場合、Unit Test では **構造的に検出不可能** である。

この乖離は Integration Test で検出する(セクション 7 を参照)。つまり:

- Unit Test でモックを使う正当性は、**Integration Test が Repository / 外部 API の実装と契約の一致を保証する** ことに依存する
- Phase 2 で Repository の Integration Test を整備するまでは、この乖離リスクを認識した上で運用する

## 6. カバレッジ方針

### 結論: 数値目標は設けない

- カバレッジ率(行・分岐・関数)は測ってもよいが、**合格ラインとして設けない**
- 数値目標は「数字を満たすためのテスト」を誘発し、意味のないテスト(getter だけ呼ぶ、分岐の片方だけ通す等)を量産する

### 代わりに問うこと

次の観点で「必要なテストが揃っているか」を判断する:

- **I/O 契約** — 公開関数の入出力が検証されているか
- **正常系** — ハッピーパス
- **異常系** — 型付きエラーが期待通り投げられるか
- **境界値** — 最小・最大・空・null
- **エラーパス** — 例外時の挙動(ロールバック等)
- **認可** — 権限違反時の拒否(該当する場合)
- **ドメインイベント** — 集約の状態変更で期待したイベントが発火しているか(該当する場合)

これらが **コードではなくレビューで担保される**。数値ではなく **「この振る舞いは守られているか」** を問い続ける。

## 7. Integration Test / E2E Test の方針

本プロジェクトにおける Integration Test / E2E Test の位置づけと判断基準を定める。整備フェーズとしては **Phase 2 以降** の対象。

### 7-1. 基本姿勢

- Integration / E2E の優先度は **Unit Test より低い**
- コスト(実行時間・保守コスト・環境構築)を考慮し、**必要な箇所に絞って** 書く
- 「書くかどうか」は、**機能の重要度** ではなく **「Unit で検出できないリスクの有無」** で判断する

### 7-2. 判断軸

Integration / E2E を書くかどうかは、以下の観点で判断する:

1. **Unit で検出できないリスクが存在するか**(最重要)
2. **失敗した時の被害がどれだけ大きいか**(非線形に大きい場合は重要度に関わらず書く)
3. **発生確率がどれだけ高いか**

「重要な機能だから書く」ではなく「Unit で保証できない領域があるから書く」というロジック。

### 7-3. Integration Test(Phase 2 で導入)

#### 必ず書く対象(コア判断と独立)

これらは **Unit Test のモック戦略の正当性を担保するために必須**:

- **Repository の実 DB Integration Test**
  - 実 DB または testcontainers を用いて I/O 契約を検証
  - Drizzle ↔ 実 PostgreSQL の整合、マイグレーション適用後の動作
  - 一意制約違反、外部キー制約、トランザクション境界などの実挙動を確認
  - これを書かないと「Unit でモックしていた契約が実装と違っていた」が本番で発覚する

#### 書くことを推奨する対象(必要性が出てきたら段階導入)

- **外部 API クライアントの Integration**
  - 実通信、または録画(VCR 的手法)による検証
  - エラーレスポンスの型、タイムアウト挙動など
- **認証・認可の境界**
  - セキュリティインシデントは重要度と独立に不可逆な被害を生むため、Integration レベルでの検証が望ましい
- **複数集約をまたぐトランザクション境界**
  - ロールバックの実挙動、分離レベルの影響など
- **決済・課金・在庫などの金銭的影響があるフロー**
  - 失敗時の被害が非線形に大きいため、モックの信頼性だけに依存させない

#### 書かなくてよい対象

- Unit で十分検証できる CRUD で、複雑なビジネスルールを持たないもの
- 管理画面の内部機能などで、失敗時の影響範囲が限定的なもの
- めったに使われないエッジ機能

### 7-4. E2E Test(原則整備しない)

現時点では **E2E Test は整備しない** 方針。理由は以下:

- Entrypoint Sociable Unit(Hono `app.request()`)で HTTP 層の正常動作は担保されている
- Repository Integration(Phase 2)で実 DB との疎通は担保される
- この 2 つが揃えば、E2E が担う「組み上げて動くか」の大部分は Unit + Integration で代替可能

#### 例外的に書く場合の対象

将来的にクリティカルパスが増えて必要性が出てきた場合のみ、以下の範囲で追加検討する:

- **スモークテスト**
  - 主要 API が起動してレスポンスを返すことだけを確認する数本
  - 「デプロイしたら全機能死んでいた」を防ぐセーフティネット
- **クリティカルパス 1〜3 本**
  - 例: サインアップ → 主要機能を 1 回実行 → ログアウト
  - 収益を生む主要ユーザーストーリーの通し確認

#### 書かない対象(例外的に整備する場合でも)

- 境界値・異常系の網羅(Unit の責務)
- 個別画面の表示詳細(Unit / コンポーネントテストの責務)
- 管理画面などクリティカルでないフロー

### 7-5. コストに関する注記

「Integration / E2E を書くコスト」だけでなく「**書かないコスト**」も天秤に入れる:

- 本番障害時の調査・復旧コスト(Unit だけではモック齟齬を検出できない)
- リファクタ時の安心感(実環境での動作保証があると内部を大きく変えられる)
- オンボーディング時の精神的保証(「このシステムは動いている」を新規参加者が信じられる)

「コスト削減のために書かない」判断をする場合は、上記の見えないコストを受け入れる判断であることを認識する。特に **Repository の Integration だけは** Unit のモック戦略の前提条件であるため、Phase 2 での整備を遅らせないこと。

## 8. エラーハンドリングのテストパターン

本プロジェクトでは typed AppError と errorMap による HTTP 変換の分離設計を採用している(詳細: `docs/server-architecture/error-handling/implementation.md`)。テストもこの分離に沿って書く。

### 層ごとの検証

| 層          | 検証                                                                             |
| ----------- | -------------------------------------------------------------------------------- |
| Domain / VO | 正しい条件で **期待した typed error が throw される** こと(`.toThrow(typeName)`) |
| errorMap    | 各エラー型が **正しい HTTP status / message に変換される** こと                  |
| Usecase     | 前提違反で **typed error が伝播する** こと(HTTP には言及しない)                  |
| Entrypoint  | HTTP レスポンスの最終形(errorMap を通じた結果)                                   |

### 原則: 二重に書かない

- HTTP status の検証は **errorMap のテスト** に閉じる
- Usecase / Domain のテストで HTTP status を検証しない(責務違反)
- VO のテストで「エラーメッセージ文字列の完全一致」を検証しない(human-readable message は errorMap 管轄)

---

## 9. 時間・乱数・環境変数

### 時間

```typescript
beforeEach(() => {
  vi.useFakeTimers();
  vi.setSystemTime(new Date("2024-01-01T00:00:00.000Z"));
});
afterEach(() => {
  vi.useRealTimers();
});
```

- `createdAt` / `updatedAt` や TTL 計算を含むテストでは必須
- 実時間に依存して flaky(稀に落ちる)状態を作らない

### 乱数 / UUID

- モックで固定化する、または「正規表現でフォーマット確認」にとどめる
- 「UUID が生成される」という振る舞いは検証してよいが、「特定の UUID 値になる」ことは検証しない

### 環境変数

- テストはデフォルトで環境変数に依存しない
- 依存する場合は `vi.stubEnv(...)` で注入・復元

---

## 10. テスト命名・配置規約

### 配置

`{FuncName}/index.ts` に対して同ディレクトリに `{FuncName}/index.test.ts` を配置する(プロジェクト全体のファイル配置規約と同じ)。

```text
createUser/
├── index.ts
└── index.test.ts
```

### 命名

- `describe` は **対象(関数名 or 概念)**
- `it` は **振る舞い(日本語文)**

```typescript
describe("createUser", () => {
  describe("正常系", () => {
    it("有効な入力で User を生成する", () => {
      /* ... */
    });
  });
  describe("異常系", () => {
    it("email が不正なら InvalidEmailFormatError を投げる", () => {
      /* ... */
    });
  });
});
```

振る舞いを日本語で書く理由:

- 英語圏のプロダクト名でない場合、日本語の方が実装者・レビュアーともに早く読める
- テスト名が自然言語の仕様になり、ドキュメント兼用になる

---

## 11. テストをいつ書くか

### 原則

- **最小単位から組み上げる実装に沿って、下層から順にテストを書く**
- Value Object → Entity / Aggregate → Policy → Usecase → Entrypoint の順に積み上げる
- Integration Test(Phase 2)は Repository を書いたら併せて書く(Unit のモック戦略の前提条件であるため)

### TDD は強制しない

- 先にテストを書くか、先に実装を書くかは個人の選好
- ただし **PR 提出時点でテストが揃っていない** のは責務放棄とみなす

### 「書くのが難しい」が出たら

- 実装を一度寝かせて設計に戻る
- テストが書けない = 疎結合ではない、のサインと解釈する
- 設計を修正してからテストに戻ると、たいていスッと書ける

---

## 12. アンチパターン

### 12-1. 内部実装を検証しているテスト

```typescript
// ❌ 内部の private 関数の呼び出し回数を検証 → リファクタで即壊れる
expect(privateFormatter).toHaveBeenCalledTimes(3);

// ✅ 公開関数の I/O だけ検証
expect(result.formattedName).toBe("Alice (admin)");
```

### 12-2. 下層の詳細を上層で再検証

```typescript
// ❌ Usecase のテストで email の形式を検証(VO の責務)
it("usecase", () => {
  expect(() => createUserUseCase({ email: "invalid" })).toThrow(/email/);
});

// ✅ Usecase では VO でエラーになることだけ確認(詳細は VO のテスト)
it("usecase", () => {
  expect(() => createUserUseCase({ email: "invalid" })).toThrow(
    InvalidEmailFormatError
  );
});
```

### 12-3. 時間・乱数・ネットワークへの直接依存

- `await fetch(...)` を直接呼ぶ Unit Test
- `new Date()` で現在時刻に依存するテスト
- 環境変数の実値に依存するテスト

すべてモックで切る。

### 12-4. モックが実装詳細を知りすぎている

```typescript
// ❌ 「内部で 2 回呼ばれる」を前提にモック
repository.find.mockResolvedValueOnce(a).mockResolvedValueOnce(b);

// ✅ 振る舞いで検証(2 回呼ばれる仕様がそもそも契約なら別だが、通常は不要)
```

### 12-5. 数値カバレッジを埋めるためだけのテスト

```typescript
// ❌ 呼ぶだけで何も検証していない
it("getter", () => {
  user.getId();
});
```

カバレッジが増えるが「振る舞いが守られている」証拠にはならない。

### 12-6. ピラミッドの逆転

- Integration Test でビジネスロジックの分岐網羅を試みる
- E2E で境界値・異常系を確認する
- Unit で書ける検証を Integration / E2E に持ち上げる

→ 遅く壊れやすいテスト群になる。**Unit で書けるものは Unit で書く。**

### 12-7. Repository をモックだけで済ませ続ける(Phase 2 以降)

- Repository の Integration Test を書かず、Unit でのモックだけに依存し続ける
- 結果、「モックしていた契約」と「実装の契約」のズレが本番で発覚する

→ Unit のモック戦略は、Repository の Integration とセットで初めて正当化される。Phase 2 での整備を後回しにしない。

## 13. まとめ

| 観点               | 方針                                                                     |
| ------------------ | ------------------------------------------------------------------------ |
| テストの目的       | I/O 契約の担保。内部実装変更への耐性                                     |
| 前提条件           | アプリケーション全体が疎結合で、各レイヤーが意味のある単位               |
| ピラミッド         | Unit 厚く / Integration は境界の疎通(Phase 2) / E2E は原則不要           |
| 構造               | 最小単位から積み上げる(Atomic Design 類推)                               |
| レイヤー責務       | レイヤーごとに振る舞いが違う → 検証対象も違う                            |
| カバレッジ         | 数値目標は設けない。振る舞い(I/O 契約)の網羅で判断                       |
| モック             | テストデータは基本モック。境界の契約をモック、内部呼び出しはしない       |
| Integration        | Repository の実 DB 検証は Phase 2 で必須。モック戦略の正当化のため       |
| E2E                | 原則整備しない。Entrypoint Sociable Unit + Repository Integration で代替 |
| 判断軸             | 「コアか」ではなく「Unit で検出できないリスクがあるか」で判断            |
| エラーハンドリング | typed error の throw 検証 / HTTP 変換は errorMap 側に閉じる              |
| ドメインイベント   | 集約ルートのテストで発火内容を契約として検証                             |
| テスト命名・配置   | `{FuncName}/index.test.ts` / 日本語 describe・it                         |
| 書くタイミング     | 下層から積み上げ。TDD 強制はしないが、PR 時点で揃っていること            |

### 補足

本ドキュメントは方針である。**具体的なコード例は `docs/testing/guidelines.md`**(別途整備)、**チェックリストは `docs/templates/test-cases.md`**(別途整備)を参照。
