# Phase 0: サービスポジショニングと運営戦略

## このドキュメントの目的

`phase0-flow-design.md` で動線とプロフィール項目の初稿を作成したが、その前段である
**「このサービスはなぜ存在するか / なぜここに来るのか」が空欄のまま**
動線設計に進んでいることが分かった。

動線・機能・項目を詰める前に、

- コンテンツとしての優位性をどこに置くか
- 運営者として継続的に何を供給するか
- それを踏まえてプロダクトはどの種類になるのか

を確定させる必要がある。本ドキュメントはその整理を行う。

> **位置づけ**
> `phase0-flow-design.md` の上位に位置するポジショニング定義。
> 本ドキュメントの結論を受けて、動線設計とプロフィール項目を改訂する。

---

## 1. 現ドラフトのコンテンツ優位性に関する懸念

`phase0-flow-design.md` の方針（外部 SNS への送り出しモデル + プロフィール公開）を
事業視点で見直したときに浮かぶ 4 つの懸念。

### 1-1. 送り出しモデルが retention を持たない

- 「応援する」を外部 SNS に逃す設計だと、訪問者が再訪する理由がない
- プレイヤー側にとっても Linktree / Lit.link 等で代替可能な機能しか提供できていない
- プロダクト内の滞留点（ここに来る理由）が空白

### 1-2. Discovery の死

- 写真 + 名前 + タグライン + ジャンルの一覧は、ランキング・キュレーション・パーソナライズがないと「知らない人の名簿」にしかならない
- Spotify / TikTok のアルゴリズム流入と比較して、ここで一覧を眺める動機が成立しない

### 1-3. 自己申告コンテンツの薄さ

- バイオ・活動コンセプトは本人が書く以上、表現にバラつきがあり情報量も乏しくなりがち
- 「このアーティストを信じていい」と思わせる第三者シグナル（出演履歴・コラボ履歴・取材記事 等）がフィールドに存在しない

### 1-4. 訪問者起点の価値が未定義

- 設計が「プレイヤーが情報を載せる」軸で進んでおり、「訪問者がここに来る理由」が言語化されていない
- 優位性は通常、(a) 独自のキュレーション、(b) 他にない情報の集約、(c) 特定シーン/コミュニティの濃さ、のいずれかから生まれるが、いまのドラフトはどれにも明確に乗っていない

## 2. retention を生むコンテンツ軸の候補

ユーザー一覧 + ユーザー詳細だけでは「箱」であって「コンテンツ」ではない。
retention を狙うには別の軸を重ねる必要がある。

| 軸         | 性質                                     | retention の効き方                       | 継続コスト       |
| ---------- | ---------------------------------------- | ---------------------------------------- | ---------------- |
| **時間軸** | イベント / 新譜 / ライブレポート         | 「定期的に新しい情報がある」で再訪が発生 | 中（運用継続）   |
| **関係軸** | コラボ / シーン / 「A が好きなら B」     | 一覧/詳細を回遊させる構造の moat になる  | 中（データ設計） |
| **編集軸** | シーン特集 / インタビュー / プレイリスト | 「ここでしか読めない」で来訪動機を作る   | 高（編集体制）   |
| **行動軸** | お気に入り / リスト / コメント           | 訪問者にアカウントを持たせて滞留         | 中（機能実装）   |

### 暫定的な優先順位

ビジョン（プレイヤーの認知拡大）と既存資産（イベント一覧/詳細のモック）を踏まえると、
**時間軸（イベント）+ 関係軸（プレイヤー × イベントの紐付け）** が最も自然な拡張。

- プレイヤー詳細から「次のライブ」「過去の出演」へ繋がり、一覧/詳細だけでは生まれない回遊と再訪動機が出る
- ただし Phase 1 のスコープを広げる判断になるため、含めるかどうかは未決

## 3. 運営者活動の必要性

事業としてみたとき、「プラットフォームを作って終わり」ではなく
**運営者が編集者/キュレーター/コミュニティ運営として継続活動する**ことが、
ほぼ必須の前提になる。

### 3-1. なぜ必要か

- **Cold start 問題**: 訪問者は「人がいない場所」に来ず、プレイヤーは「客がいない場所」に登録しない。最初の活気は運営が手で作るしかない
- **コンテンツ優位性の補完**: §1-3 の「自己申告プロフィールは薄い」を埋められるのは第三者視点で、それを最初に供給できるのは運営

### 3-2. 運営活動の種類

| 活動                                              | 効き方                                  | 継続コスト         |
| ------------------------------------------------- | --------------------------------------- | ------------------ |
| **編集**（特集 / インタビュー / シーン記事）      | 「ここでしか読めない」で来訪動機        | 高（書く力が必要） |
| **キュレーション**（今週のピック / プレイリスト） | 一覧の "意味" を作る                    | 中                 |
| **SNS 運用**（公式 X 等）                         | 流入と認知拡大の主動力                  | 中（継続性が肝）   |
| **イベント主催**                                  | プレイヤー獲得 + コンテンツ生成を兼ねる | 高（一発が重い）   |
| **アーティスト onboarding 支援**                  | 初期プレイヤー数の確保                  | 中（前期集中）     |

## 4. プロダクトの種類分岐

運営活動を前提にするかどうかで、Phase 1 以降の機能設計が大きく分岐する。

| 前提                       | プロダクトの方向                                           | 必要な機能構造                                        |
| -------------------------- | ---------------------------------------------------------- | ----------------------------------------------------- |
| 運営活動を**前提にする**   | 編集メディア型（プレイヤー紹介を編集視点で増幅する）       | ピックアップ枠 / 特集記事 / 編集者向け CMS / SNS 連携 |
| 運営活動を**前提にしない** | ツール型（プレイヤー側の運用負荷を最小化する Linktree 系） | プロフィール CRUD / SNS リンク管理 / 解析（任意）     |

`phase0-flow-design.md` 現状は後者寄りの設計になっているが、§1 の懸念を踏まえると
**前者（編集メディア型）に寄せる必要がある**可能性が高い。

---

## 5. Phase 0 / Phase 1 設計への含意

運営活動を前提にする方向に倒した場合、以下が動線設計とプロフィール項目に影響する。

| 領域               | 現ドラフトとの差分                                                                 |
| ------------------ | ---------------------------------------------------------------------------------- |
| トップページ       | プレイヤー一覧の入口だけでなく、「特集」「今週のピック」枠を持つ                   |
| プレイヤー詳細     | 自己申告フィールドだけでなく、第三者シグナル（出演履歴 / 取材記事リンク 等）を持つ |
| 一覧の意味         | 全件並べるのではなく、運営による分類軸（シーン / テーマ）を持つ                    |
| データモデル       | プレイヤー単独ではなく、プレイヤー × イベント / プレイヤー × 記事 の関係を持つ     |
| 運営オペレーション | 記事入稿 / ピックアップ管理 / プレイヤー onboarding の運用フローが必要             |

## 6. 未決事項 / 次のアクション

### 6-1. 意思決定が必要な論点

1. **プロダクトの種類**: 編集メディア型 / ツール型 / 中間（どこに振るか）
2. **採用するコンテンツ軸**: §2 の 4 軸のうち Phase 1 に含めるもの
3. **運営者の継続供給**: 何を / どの頻度で / 誰が供給するか（運営体制の前提）
4. **Phase 1 のスコープ拡張**: イベントとの紐付けを Phase 1 に入れるか Phase 2 に回すか
