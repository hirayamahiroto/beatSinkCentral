# BeatSinkCentral — 設計の中核

> すべての設計判断（動線・項目・機能・KPI）の最上位にある決定事項を集約する。
> 判断に迷ったときの最終参照先として機能させる。

---

## 🌟 ビジョン

> **業界エコシステムの構築。アーティストが持続的に活動できる環境を整備し、提供し続けること。**

このプロダクトは「特定の機能を提供するツール」ではない。
プレイヤー・オーガナイザー・ファンが**自分の利益のために使った結果、業界全体が持続可能になる構造**を設計する。

---

## 🎯 最初の課題

> **アーティストが「知られる」状態を作ること。**

知られていなければ、応援も収益もブッキングも始まらない。
業界エコシステムの最初の入口は**認知**にある。

```mermaid
flowchart LR
    D["🔍 Discover\n知る"]
    U["💡 Understand\n活動内容を理解する"]
    S["❤️ Support\n応援する"]
    D -->|設計で自然に導く| U
    U -->|設計で自然に導く| S
```

> Understandの段階に**運営者の編集・文脈づけ**が入ることが、
> Discover → Supportのファネルが機能する条件になる。
> 自己申告プロフィールだけではUnderstandが成立せず、Supportに到達しない。

## 🧠 構造的な自己認識

| 認識          | 内容                                               |
| ------------- | -------------------------------------------------- |
| ⚠️ エゴの自覚 | 「作れるから作る」エンジニア起点を自認している     |
| ✅ 本来の順序 | 課題 → 最適な手段の選択 → 実装                     |
| ❌ 現状       | 手段（プロダクト）の選択が、検証前に固定されていた |

```mermaid
flowchart TD
    V["🌟 ビジョン\n業界エコシステムの構築"]
    C["🎯 最初の課題\nアーティストが知られる状態を作る"]
    P["🛠️ プロダクト ★現在先行中"]
    M["📝 メディア"]
    E["🎪 イベント"]
    SN["📣 SNS"]
    V --> C
    C --> P & M & E & SN
    style P fill:#ffd700,stroke:#b8860b,color:#000
```

> ★ プロダクトは手段の一つに過ぎない。
> それでもプロダクトを先行させる根拠を常に明示できる状態を保つ。

## 🏗️ 三つの戦略軸

```mermaid
flowchart LR
    subgraph own["⚙️ 自分がコントロール可能"]
        PM["🛠️ プロダクト＋メディア\n基盤設計・コンテンツ統合\nアーティストに移譲可能"]
        SN["📣 SNS\n認知・流入の設計\nプロダクトへ誘導"]
    end
    subgraph ext["🤝 外部依存（実績後に動かす）"]
        EV["🎪 イベント\nオーガナイザーとの提携\nエントリー・告知基盤を提供"]
    end
    SN -->|流入| PM
    PM -->|実績を持って提携| EV
    EV -->|データ・実績| PM
```

### 実行順序

```mermaid
flowchart LR
    S1["① 📣 SNSで認知・流入\nプロダクトへ誘導する"]
    S2["② 🛠️ プロダクトで実績蓄積\nデータ・訪問者・認知を積む"]
    S3["③ 🎪 オーガナイザー提携\n実績を持って交渉する"]
    S1 --> S2 --> S3
```

> コントロール可能な軸から先に動かすのが戦略の鉄則。
> イベント軸は実績が生まれてから。

## 👤 プレイヤーのジョブ三層

```mermaid
flowchart TD
    L1["🔵 表層：機能的ジョブ\nもっと知ってもらいたい\n→ SNSで代替可能・弱い"]
    L2["🟣 中層：感情的ジョブ\n自分の活動を正しく理解してほしい\n→ 文脈ごと届く場所に差別化余地"]
    L3["🟢 深層：社会的ジョブ\nこの業界でちゃんと生きていける\n→ 離れない理由になる・最も強い"]
    L1 --> L2 --> L3
```

| 層             | ジョブ                           | 差別化余地                 |
| -------------- | -------------------------------- | -------------------------- |
| 表層（機能的） | もっと知ってもらいたい           | 低い（SNSで代替可能）      |
| 中層（感情的） | 自分の活動を正しく理解してほしい | 高い                       |
| 深層（社会的） | この業界でちゃんと生きていける   | 最大（代替が極めて少ない） |

> 設計の力点は**中層〜深層**に置く。
> 表層だけを狙うとSNSとの機能比較に巻き込まれ、ビジョンと噛み合わない。

## 💡 インセンティブ設計

| フェーズ                        | プレイヤー                       | オーガナイザー           |
| ------------------------------- | -------------------------------- | ------------------------ |
| **Phase 1**（プロダクト内完結） | 発信コストほぼゼロ・認知の可視化 | ー                       |
| **Phase 2**（イベント連携後）   | ブッキング経由の第二収入源       | 次の施策判断の根拠データ |

> 「使ってください」とお願いするのではなく、
> **使うと自分の利益になるから使われる**設計が最も強い。

## 🔄 エコシステムの全体構造

```mermaid
flowchart TD
    PL["🎤 プレイヤー\nコンテンツ発信・認知収益を得る"]
    PR["⚙️ プロダクト\n中心インフラ\nデータ・告知・エントリー一元化"]
    OR["🎪 オーガナイザー\n告知・エントリー管理\nブッキング判断の根拠"]
    FN["👥 ファン・訪問者\n発見・応援・イベント参加\n行動データを生む"]

    PL -->|コンテンツ・注目度データ| PR
    PR -->|認知の可視化・収益機会| PL
    OR -->|告知・エントリー・イベントデータ| PR
    PR -->|ブッキング根拠・計測結果| OR
    FN -->|行動データ| PR
    PR -->|アーティスト発見・イベント情報| FN
    PL -.->|ブッキング| OR
    OR -.->|収益| PL
```

| 参加者            | 使う理由                   | 渡すもの                 | 得るもの                       |
| ----------------- | -------------------------- | ------------------------ | ------------------------------ |
| 🎤 プレイヤー     | 認知・収益のため           | コンテンツ・注目度データ | 認知の可視化・収益機会         |
| 🎪 オーガナイザー | 告知・エントリー管理のため | イベントデータ           | ブッキング根拠・計測結果       |
| 👥 ファン         | アーティストを発見するため | 行動データ               | アーティスト発見・イベント情報 |

> Phase 1では**プレイヤー ↔ ファン**の辺を最初に成立させる。
> Phase 2以降で**オーガナイザー**の辺を統合する。

## 📊 計測設計

### 認知可視化の三層

```mermaid
flowchart TD
    A["Layer 1　量（到達）\nどれだけ知られているか\nアクセス数・UU数・流入経路"]
    B["Layer 2　質（関心）\nどう知られているか\n滞在時間・回遊率・リピーター率"]
    C["Layer 3　転換（行動）\n誰がどう動いたか\n応援アクション・ユーザー種別・外部遷移率"]
    A --> B --> C
```

### Discover / Understand / Support 対応指標

| 段階       | 問い               | 指標                                 |
| ---------- | ------------------ | ------------------------------------ |
| Discover   | 知られているか     | プロフィールアクセス数・一覧露出回数 |
| Understand | どう知られているか | 平均滞在時間・スクロール深度・完読率 |
| Support    | 応援に至ったか     | SNSリンククリック・外部遷移数        |

### 計測ループ

```mermaid
flowchart TD
    A["🔍 プレイヤーの注目度が可視化される"]
    B["📋 オーガナイザーがブッキング判断"]
    C["🎪 イベント実施\n告知・エントリーもプロダクト経由"]
    D["📈 流入・満足度を計測"]
    E["⬆️ 次のブッキング精度が上がる"]
    A --> B --> C --> D --> E
    E -->|使うほど精度が上がる| A
```

## 📐 情報設計の階層

```mermaid
flowchart TD
    L1["Layer 1　戦略\nビジョン・課題・三軸・インセンティブ・エコシステム構造\n✅ 本ドキュメントで完了"]
    L2["Layer 2　情報構造\n動線設計・データ構造定義・価値の定義\n← 次に設計すべき"]
    L3["Layer 3　画面設計\nプレイヤー詳細・ダッシュボード・オーガナイザー画面・一覧\n情報構造が決まってから"]
    L1 --> L2 --> L3
```

### Layer 2で答える三つの問い

| 問い           | 内容                             | 未決の状態だと             |
| -------------- | -------------------------------- | -------------------------- |
| 動線設計       | 誰がどこから入りどこへ向かうか   | 何を作るかの判断基準がない |
| データ構造定義 | 動線上の行動を何として記録するか | Phase 2の価値が生まれない  |
| 価値の定義     | 誰のどんな判断を助けるか         | 誰にとっても中途半端になる |
