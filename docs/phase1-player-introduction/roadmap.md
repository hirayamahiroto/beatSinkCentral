# MVP Roadmap - Phase 1: プレイヤー紹介

## 概要

Phase 1 の MVP は「プレイヤー紹介機能」の実現に焦点を当てる。
ユーザーが登録し、プレイヤーとしてプロフィールを作成・公開できる状態をゴールとする。

## 現状

### 実装済み

| レイヤー | 機能 | 状態 |
|---------|------|------|
| api-server | User Entity / Value Objects / Factory / Behaviors | 済 |
| api-server | `POST /api/users/create` | 済 |
| api-server | Auth0 認証 Middleware | 済 |
| api-server | UserRepository (save, findBySub) | 済 |
| api-server | CreateUser UseCase（冪等性あり） | 済 |
| database | users テーブル | 済 |
| beatfolio | Auth0 Login / Logout | 済 |
| beatfolio | オンボーディング画面（API接続済み） | 済 |
| ui | プレイヤー一覧/詳細ページ（モックデータ） | 済 |
| ui | イベント一覧/詳細ページ（モックデータ） | 済 |
| ui | 管理画面（モックデータ） | 済 |

### 未実装（MVPに必要）

- 要件定義（ペルソナ、課題設定、提供形態の決定）
- ユーザー登録フローの完成（未登録検知・リダイレクト）
- プレイヤープロフィールのドメイン設計・API・DB
- プレイヤー一覧/詳細のDB連携（モック → 実データ）
- 公開ページの整備

## Phase 0: 要件定義

実装に入る前に、プロダクトの方向性と各機能の要件を固める。

### 解決すべき課題

- なぜ作るのか、何を解決したいか
- 課題の根本が芯を捉えているか
- ペルソナが設定できていない（誰のための機能か）
- SaaS として提供する必要があるのか、別の提供形態が適切か

### 内部的な課題

- UI 周りの設計が暫定的なため作り直す必要がある
- モックデータで作られた UI と実際の要件にギャップがある可能性

### 各ページの要件整理（未精査）

#### トップページ

- ログインなしでもページ全体を表示して良い
- ヘッダー右にログイン/サインアップの CTA を表示

#### ユーザー登録

- Auth0 認証後、dashboard へ遷移
- プロフィール入力完了でプレイヤー一覧 UI に表示される
- プロフィール内容は未精査（Phase 0 で確定する）

#### プレイヤー一覧

- プロフィール登録していないユーザーは一覧に表示されない
- 一覧表示に必要なミニマム情報:
  - 名前
  - アーティスト写真

### Phase 0 のアウトプット

- [ ] ペルソナ定義
- [ ] 解決する課題の明文化
- [ ] 提供形態の決定（SaaS / その他）
- [ ] 各ページの要件仕様書
- [ ] プロフィール項目の確定

## Phase 1: MVP 実装

### Step 1: ユーザー登録フローの完成

ログインからプロフィール作成までの一連の流れを完成させる。

```
Auth0 signup → beatfolio → 未登録検知 → /onboarding → DB保存 → dashboard
```

#### 要件

- Auth0 認証後、dashboard へ遷移する
- 未登録ユーザー（DB に User が存在しない）は `/onboarding` へリダイレクト
- オンボーディングでプロフィール入力が完了するとプレイヤー一覧に表示される
- 登録済みユーザーが `/onboarding` にアクセスした場合は dashboard へリダイレクト

#### 着手前に設計が必要な事項

- `GET /api/users/me` のレスポンス形式（どのユーザー情報を返すか）
- 未登録検知のタイミング（Middleware で行うか、ページ側で行うか）
- オンボーディングで収集する項目の確定（現状は username と accountId のみ）
- dashboard の画面構成

#### タスク

| タスク | アプリ | 詳細 |
|--------|-------|------|
| `GET /api/users/me` 実装 | api-server | セッションの sub からユーザー情報を取得するエンドポイント |
| `requireRegisteredMiddleware` 実装 | api-server | DB登録済みかチェックする Middleware |
| ログイン後の未登録リダイレクト | beatfolio | `/api/users/me` を呼び、未登録なら `/onboarding` へリダイレクト |
| 登録済みユーザーのオンボーディングスキップ | beatfolio | 登録済みなら `/onboarding` をスキップして dashboard へ |
| dashboard ページ作成 | beatfolio | 認証後のランディングページ |

### Step 2: プレイヤープロフィールのドメイン設計

MVP の核となるプレイヤーデータモデルを設計・実装する。

#### 要件

- プロフィール登録が完了したユーザーのみプレイヤー一覧に表示される
- 一覧表示に必要なミニマム情報: 名前、アーティスト写真
- プロフィール内容の詳細は Phase 0 で確定する（活動名、活動地域、スキル、経歴、SNSリンク等の取捨選択）

#### 着手前に設計が必要な事項

- Player Profile Entity のフィールド定義（Phase 0 の結果を反映）
- User と Player Profile の関係（1:1 か 1:N か）
- Value Objects の設計（どのフィールドに独自のバリデーションが必要か）
- `player_profiles` テーブルのスキーマ設計（カラム型、制約、インデックス）
- プロフィール作成・編集画面のフォーム項目とバリデーションルール
- アーティスト写真のアップロード・保存方式（S3 等）

#### タスク

| タスク | アプリ | 詳細 |
|--------|-------|------|
| Player Profile Entity 設計 | api-server | Phase 0 で確定した項目に基づく |
| `player_profiles` テーブル追加 | database | マイグレーション作成・適用 |
| プロフィール作成 API | api-server | `POST /api/players` |
| プロフィール更新 API | api-server | `POST /api/players/:id` |
| プロフィール取得 API | api-server | `GET /api/players/:id` |
| 画像アップロード機能 | api-server / infrastructure | アーティスト写真の保存 |
| プロフィール作成・編集画面 | beatfolio | フォーム UI、API 連携 |

### Step 3: プレイヤー一覧・詳細の DB 連携

UI のモックデータを実データに切り替える。

#### 要件

- プロフィール登録済みのプレイヤーのみ一覧に表示される
- 一覧にはミニマム情報（名前、アーティスト写真）を表示
- 詳細ページではプロフィールの全情報を表示

#### 着手前に設計が必要な事項

- 一覧 API の検索・フィルタ仕様（どのフィールドで絞り込むか、全文検索の要否）
- ソート条件の定義（名前順、登録日順、地域順 等）
- ページネーション方式の選定（offset-based か cursor-based か）
- 既存モック UI のデータ構造と API レスポンス形式の差分整理
- 一覧表示のパフォーマンス要件（件数が増えた場合の対策）

#### タスク

| タスク | アプリ | 詳細 |
|--------|-------|------|
| プレイヤー一覧取得 API | api-server | `GET /api/players`（検索・フィルタ・ソート・ページネーション） |
| プレイヤー詳細取得 API | api-server | `GET /api/players/:id`（公開プロフィール） |
| 一覧ページの API 連携 | beatfolio | モックデータ → API 呼び出しに切り替え |
| 詳細ページの API 連携 | beatfolio | モックデータ → API 呼び出しに切り替え |

### Step 4: 公開ページの整備

未ログインユーザーでもプレイヤー情報を閲覧できるようにする。

#### 要件

- トップページはログインなしでもページ全体を表示して良い
- ヘッダー右にログイン / サインアップの CTA を表示
- プレイヤー一覧・詳細は未ログインでも閲覧可能

#### 着手前に設計が必要な事項

- 認証不要で公開する API エンドポイントの範囲（一覧・詳細どこまで公開するか）
- 公開情報と非公開情報の境界（プロフィールのどの項目を未ログインユーザーに見せるか）
- SEO 要件の整理（OGP に含める情報、動的メタデータの生成方式）

#### タスク

| タスク | アプリ | 詳細 |
|--------|-------|------|
| 公開 API（認証不要）の整備 | api-server | プレイヤー一覧・詳細を認証なしで取得可能にする |
| 未ログイン時の UI 調整 | beatfolio | ヘッダー、CTA（ログイン / サインアップボタン）の表示切り替え |
| SEO 対応 | beatfolio | メタデータ、OGP 設定 |

## 優先順位

```
Phase 0（要件定義）
  ↓  要件が固まらないと実装の判断基準がない
Step 1（ユーザー登録フロー）
  ↓  ユーザーが DB に存在しないと後続が成り立たない
Step 2（プレイヤープロフィール設計）
  ↓  データモデルがないと API・UI が作れない
Step 3（DB 連携）
  ↓  実データで動くプレイヤー紹介
Step 4（公開ページ）
     MVP として外部公開できる状態
```
