# Phase 0: 動線設計 — プレイヤー紹介

## このドキュメントの目的

Phase 1（プレイヤー紹介機能）の実装に入る前に、プロフィール項目の列挙ではなく
**動線（ユーザーがどこから来て・どう進み・どこへ抜けるか）** を先に確定させる。

動線が決まることで、

- 各ページに**何を載せるべきか**（プロフィール項目）
- どの API を**いつまでに作るべきか**（実装順序）
- どこまでを Phase 1 のスコープに含めるか

が逆算できるようになる。本ドキュメントは Phase 1 実装計画（Step 1〜Step 4）の上位設計として位置づける。

---

## 1. 前提（roadmap.md からの引き継ぎ）

| 項目           | 内容                                                   |
| -------------- | ------------------------------------------------------ |
| ビジョン       | プロダクト上のプレイヤーの認知拡大                     |
| ファネル       | 知る → 理解する → 応援する →（再訪）                   |
| 第一ペルソナ   | 一般リスナー / 潜在ファン（閲覧者）                    |
| 対象外ペルソナ | イベント主催者・同業プレイヤー等は Phase 2 以降        |
| 設計アプローチ | プロフィール項目の列挙ではなく、動線設計を成果物にする |

## 2. 確定事項（Step 0-A）

roadmap.md の「先に確定すべき意思決定」3 つに対する Phase 1 の方針。

### 2-1. 「応援する」の到達地点

**決定: 外部プラットフォームへ送り出す（B 案）**

- フォロー機能・通知機能を Phase 1 で持たない
- プレイヤーが既に運用している SNS / 配信サービスへ遷移させることで「応援する」を成立させる
- 理由:
  - 実装コスト最小で Phase 1 のスコープに収まる
  - プレイヤー側の運用負荷も最小（既存 SNS をそのまま使える）
  - 自プロダクトでフォローを抱える前に、まず「認知拡大」が機能するかを検証する

### 2-2. 再訪導線

**決定: Phase 1 のスコープに含めない**

- Phase 1 のゴールは「単方向ファネル（知る → 応援する）が成立すること」
- 通知・フォロー・更新検知は Phase 2 以降
- 理由:
  - 再訪導線は「応援する」の到達地点（2-1）と密接に絡むため、まず一方向の流れを完成させる
  - 再訪の成功指標を立てる前段として、知る/理解する/応援する各段階の通過率を観測したい

### 2-3. 入口の範囲

**決定: SEO + OGP + トップページ + 一覧ページ を最小セットとする**

- SEO: プレイヤー詳細ページが検索結果に出る状態を作る
- OGP: SNS シェア時にカード表示される状態を作る
- トップページ: 直接訪問の受け皿
- 一覧ページ: トップから/外部からの両方で導線になる
- スコープ外:
  - SNS シェアボタン（プレイヤー側ではなく閲覧者の自発的シェアに依存）
  - サイト内検索 / カテゴリページ
  - レコメンド機能

## 3. 動線マトリクス（Step 0-B）

4 区間 × 6 要素で動線を定義する。`再訪` は Phase 1 スコープ外のため対象外。

### 3-1. 入口（SEO / OGP / トップ / 一覧）

| 要素           | 内容                                                                                  |
| -------------- | ------------------------------------------------------------------------------------- |
| 入口           | Google 検索 / SNS でのシェアからの流入 / プロダクト URL の直接訪問                    |
| 表示情報       | 検索結果のタイトル・概要 / OGP カード（プレイヤー名・写真・タグライン）/ トップページ |
| ユーザー行動   | クリック / リンクタップ / トップを開いて一覧 or 詳細へ進む                            |
| 次への遷移条件 | プレイヤー名 or アーティスト写真をクリックして「知る」へ                              |
| 離脱の許容     | 許容（流入元が広いため、ターゲット外の離脱は当然発生する）                            |
| 完了判定       | 一覧ページ or プレイヤー詳細ページに到達したか                                        |

### 3-2. 知る（一覧 / トップ）

| 要素           | 内容                                                         |
| -------------- | ------------------------------------------------------------ |
| 入口           | 入口区間（SEO / OGP / トップ）からの遷移                     |
| 表示情報       | アーティスト写真 / 名前 / タグライン / ジャンル              |
| ユーザー行動   | 興味を持ったプレイヤーをタップして詳細へ進む                 |
| 次への遷移条件 | プレイヤーカードのクリック → 詳細ページへ                    |
| 離脱の許容     | 許容（一覧で誰にも興味を持たれなかった場合の離脱は仕方ない） |
| 完了判定       | 1 人でもプレイヤー詳細ページへ遷移したか                     |

### 3-3. 理解する（プレイヤー詳細）

| 要素           | 内容                                                                                   |
| -------------- | -------------------------------------------------------------------------------------- |
| 入口           | 一覧からの遷移 / SEO・OGP からの直接流入                                               |
| 表示情報       | アーティスト写真 / 名前 / タグライン / ジャンル / バイオ / 活動コンセプト / SNS リンク |
| ユーザー行動   | 経歴・活動内容を読む / SNS リンクを認識する                                            |
| 次への遷移条件 | SNS リンクのクリック（応援するへの遷移）                                               |
| 離脱の許容     | 一部許容（応援する到達は理想だが、内容を読んで離脱でも一定の認知拡大目的は満たす）     |
| 完了判定       | SNS リンクが 1 つ以上クリックされたか / 詳細ページの平均閲覧時間                       |

### 3-4. 応援する（外部 SNS / 配信サービスへの送り出し）

| 要素           | 内容                                                                 |
| -------------- | -------------------------------------------------------------------- |
| 入口           | プレイヤー詳細ページの SNS リンク                                    |
| 表示情報       | （自プロダクトでは表示しない。外部プラットフォーム側に委ねる）       |
| ユーザー行動   | 外部 SNS でのフォロー / 配信サービスでのフォロー / 楽曲再生          |
| 次への遷移条件 | （Phase 1 では再訪導線を持たないため、ここがファネルの終点）         |
| 離脱の許容     | 完全に許容（自プロダクトを離脱して外部で応援が成立すれば成功）       |
| 完了判定       | SNS リンクの外部遷移クリック数（自プロダクト側で計測できる最終指標） |

## 4. プロフィール項目（Step 0-C）

動線から逆算して、Phase 1 で必須となるフィールドを確定する。

### 4-1. フィールド一覧

| フィールド名     | 用途（どの動線区間で使うか） | 型             | 必須/任意         | 公開範囲 | 備考                                                        |
| ---------------- | ---------------------------- | -------------- | ----------------- | -------- | ----------------------------------------------------------- |
| 名前（活動名）   | 入口（OGP）/ 知る / 理解する | string         | 必須              | 全体公開 | 表示上の名前。`username` とは別のフィールドとして持つ       |
| アーティスト写真 | 入口（OGP）/ 知る / 理解する | URL（画像）    | 必須              | 全体公開 | 一覧サムネイルと詳細ページ両方で使う                        |
| タグライン       | 入口（OGP）/ 知る / 理解する | string（短文） | 必須              | 全体公開 | 1〜2 行で活動を端的に表す。OGP description にも使用         |
| ジャンル         | 知る / 理解する              | string（複数） | 必須              | 全体公開 | 一覧でのフィルタ・詳細での文脈情報                          |
| バイオ           | 理解する                     | text           | 必須              | 全体公開 | 経歴・活動歴。詳細ページでのみ表示                          |
| 活動コンセプト   | 理解する                     | text           | 必須              | 全体公開 | バイオとは別に「何を目指しているか」を伝える項目            |
| SNS リンク       | 応援する                     | URL（複数）    | 必須（最低 1 つ） | 全体公開 | Phase 1 のファネル終点。最低 1 つ登録しないと公開不可とする |

### 4-2. Phase 1 では持たないフィールド

動線上 Phase 1 では未使用のため、データモデルにも含めない。Phase 2 以降で再検討する。

| フィールド              | 理由                                       |
| ----------------------- | ------------------------------------------ |
| カバー画像              | 「知る」「理解する」のミニマムには不要     |
| 動画サムネイル          | Phase 2（理解するの拡張）                  |
| 詳細経歴 / メディア掲載 | Phase 2（理解するの拡張）                  |
| 楽曲埋め込み            | 外部 SNS / 配信に送り出すため Phase 1 不要 |
| 出演予定                | 再訪導線とセット。Phase 2                  |
| フォロー機能            | 自プロダクト内応援は Phase 2               |

### 4-3. 補足：登録必須条件

「プロフィール登録済み = プレイヤー一覧に表示される」状態を成立させるため、上記必須フィールドが**全て埋まっている**ことを「登録完了」の条件とする。
途中保存を許容するか、一括入力にするかは UI 設計（Phase 1 Step 2）で決める。

本ドキュメントの結果は roadmap.md の各 Step に以下の形で反映する。

## 6. 未決事項（Phase 1 着手前にさらに詰める）

本ドキュメントで意図的に未決にしている項目。Step 1〜Step 4 着手時に確定する。

- 「ジャンル」のマスタを固定リストにするか自由入力にするか
- アーティスト写真のアップロード方式（直接アップロード / 外部 URL 指定）
- タグラインの最大文字数（OGP 表示と詳細ページ表示の両立）
- 「途中保存」を許容するか、一括入力フォームにするか
- 公開を停止する手段（プロフィール非公開フラグの要否）
