# Phase 0: サービスの根幹設計

## このドキュメントの目的

`phase0-service-positioning.md` で「コンテンツ優位性が薄い」「運営活動が必要」という
論点まで詰めたが、それらを判断する**基準そのもの**が空欄だった。

動線設計（`phase0-flow-design.md`）も、ポジショニング（`phase0-service-positioning.md`）も、
すべての下位設計は **「このサービスは何であるか」** という根幹から派生する。

本書では下位設計が判断不能なまま走ることを防ぐため、
**「領域 × 編集視点 × 三者利得」を 1 文に圧縮する** ことを成果物とする。

> **位置づけ**
> 既存の Phase 0 ドキュメントの最上位に置く。本書の結論をもとに
> `phase0-service-positioning.md` → `phase0-flow-design.md` の順で改訂する。

## 1. なぜ「根幹設計」を独立して扱うか

ここまでの議論を辿ると、判断が必要な順序は次のようになっている:

```
[根幹]  ← サービスは何であるか / なぜここでなければならないか
   ↓
[ポジショニング]  ← 編集メディア型 / ツール型 のどちらに寄せるか
   ↓
[動線・コンテンツ軸]  ← 何を見せ、どう回遊させ、どこで応援に至らせるか
   ↓
[項目・機能・実装]  ← フィールド / API / UI
```

下位の設計（動線、項目、機能）は、上位（根幹）が決まらないとすべて**等価に正しく**見えてしまう。
たとえば「タグラインを必須にすべきか」「イベントを Phase 1 に入れるべきか」は、
**「どんなサービスでありたいか」**が決まって初めて正解が出る問いである。

## 2. 根幹を構成する 3 要素

### 2-1. 領域 / シーン

**問い**: どの音楽シーン・地域・カルチャーを対象にするか。

- 「音楽全般」を対象にすると、誰のためのサービスかが曖昧になり Discovery が機能しない
- シーンを絞ることで、一覧に載る "誰" の基準が定義でき、第三者視点（編集）も成立する
- 絞り方は「ジャンル」「地域」「世代」「活動形態」「カルチャー文脈」などの軸の組み合わせ

**判断基準**:

- 運営者自身が体感として「いい」「悪い」を判断できる範囲か
- 5 年後も対象にしたいと言えるか（短期トレンドではないか）
- そのシーンに既存の場（メディア・コミュニティ）がどれだけあるか / ないか

### 2-2. 編集視点 / 価値判断

**問い**: 何を「いい」と定義し、何を載せ / 何を載せないか。

- 自己申告型のプロフィール集に陥らないために、運営者の **価値判断軸** が必要
- 「全員載せる」プロダクトは編集視点を持たないため、コンテンツ優位性が生まれない
- 編集視点 = 「ここに載っていることそのものが意味を持つ」状態を作る基準

**判断基準**:

- 1 文で言語化できるか（例: 「自分の言葉で曲を語れるアーティストだけ」）
- 載せない理由を運営者が説明できるか
- 訪問者がその基準を体感したときに「だからここを見るのか」と納得できるか

### 2-3. 三者の利得

**問い**: 訪問者 / プレイヤー / 運営者がそれぞれここから何を得るか。

| 主体           | 問い                                                  | 答えがないと起きること                      |
| -------------- | ----------------------------------------------------- | ------------------------------------------- |
| **訪問者**     | ここに来ることでしか得られない体験は何か              | 既存サービス（Spotify / TikTok 等）に流れる |
| **プレイヤー** | ここに載ることで他では得られない利得は何か            | Linktree 等で代替され、登録動機が生まれない |
| **運営者**     | これを続けることで何を実現したいか / どう持続させるか | 編集・運営の継続意思が折れる                |

3 主体すべてに「なぜここでなければならないか」が同時に成り立つことが、
根幹が**機能している**最低条件。1 つでも欠けるとサービス全体が崩れる。

## 3. 領域 × 編集視点の組み合わせ例

具体例があると判断しやすいため、成立しうるパターンを列挙する（採用案ではない）。

| 領域                                       | 編集視点                                 | 結果として顔が変わる例                          |
| ------------------------------------------ | ---------------------------------------- | ----------------------------------------------- |
| 日本のアンダーグラウンド HipHop シーン     | ライブ現場視点で記録する                 | 出演履歴 / 取材レポートが第一義のメディア型     |
| 地方の若手バンド                           | ジャンル横断のシーンとして編む           | 地域 + ジャンルクロスのキュレーションメディア型 |
| ビートメイカー × MC                        | コラボ生成を支援する                     | マッチング機能を持つツール × ネットワーク型     |
| インディペンデント運営の音楽家             | DIY な活動の記録として残す               | アーカイブ + ストーリーテリング型               |
| 特定の年代・カルチャー（例: Gen Z シーン） | 同世代の感覚で「いま聴くべき」を定義する | 編集色の濃いキュレーション型                    |

同じ「プレイヤー紹介」でも、組み合わせ次第で
**必要なフィールド・動線・運営活動・データモデルがまったく別物**になる。

## 4. 根幹を圧縮した 1 文（テンプレート）

最終アウトプットは次の 1 文を埋めること。

```
[領域・シーン] のプレイヤーを、
[編集視点・価値判断] の観点で取り上げ、
[訪問者の利得] / [プレイヤーの利得] / [運営者として目指す状態] を実現するサービス。
```

例（あくまで形式の例示）:

> 日本のアンダーグラウンド HipHop シーンのプレイヤーを、
> ライブ現場での実績と語りの説得力という観点で取り上げ、
> リスナーには "次に行くべきライブ" を、プレイヤーには "現場での実績が可視化される場" を、
> 運営者には "シーンのアーカイブを残す活動" を実現するサービス。

---

## 5. 根幹の検証チェックリスト

1 文ができたら、以下のテストに通るかを確認する。

- [ ] **代替テスト**: 既存サービス（Spotify / TikTok / Instagram / Linktree 等）で同じ価値が得られないか
- [ ] **第三者説明テスト**: 「新人に 1 文で説明できるか」「友人に何のサービスか伝えられるか」
- [ ] **編集判断テスト**: 「このプレイヤーを載せるか / 載せないか」を運営者が即答できるか
- [ ] **三者利得テスト**: 訪問者・プレイヤー・運営者の 3 主体すべてに「なぜここか」が言えるか
- [ ] **5 年テスト**: この方針で 5 年運営し続けられるか（短期トレンド依存ではないか）
- [ ] **逆引きテスト**: この 1 文から動線 / プロフィール項目 / 機能の優先順位が逆算できるか

1 つでも通らない場合、根幹がまだ煮詰まっていない可能性が高い。

## 参考になる著書

ここで扱っている「サービスの根幹」「ポジショニング」「コールドスタート」「編集メディア性」を
体系的に学ぶうえで参考になる書籍を、論点ごとに整理する。

### 「なぜここでなければならないか」を言語化する

- **『ストーリーとしての競争戦略』楠木建** — 「一見非合理に見える選択が物語として繋がっているか」という競争戦略の捉え方。1 文に圧縮する作業の精度を上げる
- **『Obviously Awesome』April Dunford** — ポジショニングを実務的なフレームに落とし込んだ本。"who is this for / what is it / why is it different" を埋める手順がそのまま使える
- **『Positioning』Al Ries & Jack Trout** — ポジショニングの古典。「マインドの中に居場所を作る」という発想

### 顧客起点で「利得」を定義する

- **『ジョブ理論』 Clayton Christensen（原題: Competing Against Luck）** — 「顧客は製品を雇う」という視点。三者利得（訪問者 / プレイヤー / 運営者）を JTBD で書き直すと精度が上がる
- **『Inspired』Marty Cagan** — プロダクトマネジメントの土台。価値（valuable）/ 実現可能性（feasible）/ 使いやすさ（usable）/ 事業性（viable）の 4 リスクで設計を点検する

### コールドスタート / ネットワーク効果

- **『The Cold Start Problem』Andrew Chen** — まさに本ドキュメントの懸念（cold start / 運営活動の必要性）と同じ問いに正面から答える本。"Atomic Network" の考え方は、領域・シーンの絞り込みの根拠になる
- **『Platform Revolution』Geoffrey Parker, Marshall Van Alstyne, Sangeet Choudary** — プラットフォーム / マルチサイドの設計。プレイヤーと訪問者の双方価値をどう成立させるかの参考

### コンテンツ / カルチャー / 編集の moat

- **『Cultural Strategy』Douglas Holt** — カルチャーを moat として扱うブランド戦略論。編集視点・シーン定義の根拠を文化的文脈から取る発想
- **『Tribes』Seth Godin** — コミュニティ / 部族としてのユーザー像。編集軸とコミュニティ運営をつなぐ視点

### イシュー / 問題設定

- **『イシューからはじめよ』安宅和人** — 「解くべき問いの設定」が解の質を決めるという思想。本書の §1〜§2 のような上流の問い直しに有効

### 補足

- 上記はすべて「下位の設計を始める前に上流を疑う」ための参考。読み切ることが目的ではなく、判断に詰まったときに 1 章だけ拾い読みする使い方が現実的
- 音楽産業 / シーン論に直接触れる本としては学術書 / 評論が中心になり、プロダクト設計に直結する一冊は少ない。むしろ既存メディア（例: ele-king / Mikiki / FNMNL 等）の編集方針を観察するほうが、編集視点の参考になる場合がある
